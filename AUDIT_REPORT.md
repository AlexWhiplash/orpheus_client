## Фаза 0 — Разведка (стек, масштаб, диагностика)

**Что за продукт.** Мобильный (Android-first) защищённый мессенджер: сквозное шифрование
(X25519 + ChaCha20-Poly1305), звонки WebRTC с нативным UI входящего звонка (CallKit),
real-time через WebSocket с HTTP-fallback, push через Firebase/FCM, PIN/duress-режим/panic-wipe,
AI-ассистент «Oracle of Orpheus», групповые комнаты. Рынок — русскоязычный. Часть
мульти-компонентного workspace (клиент/десктоп/бэкенд/relay/сайт); аудируется только клиент.

**Стек.** Flutter 3.44.4 (stable), Dart 3.0+; `pubspec` version 1.1.6+12. Ключевые пакеты:
`cryptography`, `flutter_secure_storage`, `sqflite`, `firebase_core/messaging`, `flutter_webrtc`,
`flutter_callkit_incoming`, `web_socket_channel`, `provider`, `sentry_flutter`, `dio` + `http`.

**Масштаб.** `lib/`: 87 файлов Dart, ~26 200 строк (без сгенерированного l10n; с ним ~32 400).

## Сводка

Orpheus — на удивление зрелая бета: полноценное E2E-шифрование, WebRTC-звонки с нативным UI,
AI-ассистент, комнаты, зашифрованный блокнот, богатый набор безопасности (PIN/duress/panic-wipe) и
чистая локализация EN/RU. **Крипто-примитивы применены грамотно** (ChaCha20-Poly1305 со случайным
nonce, проверка MAC, ключи только в Keystore, шифрование в изолятах, параметризованный SQL, `wss/https`
без обхода сертификатов). **Но главная проблема — не в примитивах, а в дизайне хранения и приватности:**
вся переписка, контакты и заметки лежат на устройстве в **незашифрованном** SQLite (сообщения
расшифровываются перед сохранением), поэтому PIN, duress-режим и wipe — это косметические экранные
заслонки, которые криминалистический инструмент обходит. Плюс телеметрия по умолчанию шлёт на сервер
публичные ключи контактов и отпечаток устройства, у схемы нет forward secrecy, а release-сборка
подписана публичным debug-ключом. Есть и пара реальных багов доставки (тихая потеря сообщений, «удалить
у обоих» не срабатывает у получателя). Итог: **прочный фундамент, но перед публичным релизом обязательны
несколько исправлений уровня «безопасность/данные»** — они перечислены в плане действий.

Всего находок: **105** по всем 13 фазам (проверено по коду; критичные крипто/at-rest выводы аудитор
перепроверил независимо и подтвердил). Распределение: 🔴 1 критично · 🟠 23 высокий · 🟡 45 средний ·
🟢 36 низкий (часть — сквозные, всплывшие в нескольких фазах; ниже сведены).

## Находки

### 🔴 Критично (утечка данных / потеря данных / падение прода)

**[SEC-1 / DB-1] Локальная база данных хранится в открытом виде (нет шифрования на устройстве)** — `lib/services/database_service.dart:41`
- **Что не так:** `openDatabase()` создаёт обычный файл `orpheus.db` без SQLCipher/пароля. Входящие
  сообщения расшифровываются *до* сохранения (`incoming_message_handler.dart:288-297` пишет
  `text: decryptedMessage`), исходящие сохраняются как введённый текст (`chat_screen.dart:108-116`).
  Контакты, заметки и AI-переписка — тоже открытые колонки.
- **Чем грозит (простым языком):** Весь смысл защищённого мессенджера — чтобы переписка была
  недоступна, если телефон попал в чужие руки. Здесь же вся история чатов, контакты и приватные заметки
  лежат на телефоне обычным читаемым текстом. Любой, кто получил разблокированный (или рутованный)
  телефон, снял копию хранилища или бэкап, читает всё. **PIN, «пустой» duress-профиль и авто-wipe только
  прячут данные внутри экранов приложения — они не шифруют то, что записано на диск**, поэтому
  криминалистический инструмент их игнорирует. Это подрывает главное обещание продукта.
- **Как чинить:** Зашифровать БД через SQLCipher (`sqflite_sqlcipher` / `sqlcipher_flutter_libs`)
  ключом из `flutter_secure_storage` (Keystore) и/или производным от PIN через memory-hard KDF (Argon2id),
  чтобы PIN реально защищал данные. Ключ БД стирать при panic-wipe. Это самое важное изменение уровня данных.
- **Трудозатраты:** L

### 🟠 Высокий приоритет (баги и риски — чинить в ближайший спринт)

**[SEC-2] Телеметрия по умолчанию шлёт на сервер публичные ключи контактов и отпечаток устройства** — `lib/services/telemetry_service.dart:28`
- **Что не так:** `TelemetryService` включён по умолчанию (`_enabled = true`, старт в `main.dart:178`),
  подписывается на каждый лог `DebugLogger` и POST-ит батчи на `/api/logs/batch` с заголовком
  `X-Pubkey` = ключ пользователя. В нагрузку (`_serializeEntry`, стр. 77-94) входят `peer_pubkey`,
  `call_id`, модель/ОС устройства, состояние сети. Экрана согласия и отключения в коде нет.
- **Чем грозит:** Даже если текст сообщений зашифрован, наружу тихо уходят именно те метаданные, которые
  деанонимизируют человека — **с кем** он переписывается и звонит, **когда** и с какого телефона, — со
  штампом его ключа-личности. Тот, кто владеет сервером (или может его принудить), восстанавливает граф
  контактов и отпечаток устройства. Для приложения, которое продаётся на анонимности, это прямое
  противоречие собственному обещанию.
- **Как чинить:** Сделать телеметрию строго opt-in (по умолчанию выключено) с понятным экраном согласия;
  убрать из выгрузки `peer_pubkey`/идентификаторы контактов и отпечаток устройства; добавить реальный
  выключатель; задокументировать, что именно собирается.
- **Трудозатраты:** M

**[SEC-3 / COMP-1] Нет forward secrecy — утечка одного приватного ключа раскрывает всю переписку** — `lib/services/crypto_service.dart:167`
- **Что не так:** Шифрование использует статический долговременный ключ X25519 с обеих сторон
  (стр. 167-174 / 203-216). `sharedSecretKey()` детерминирован → один и тот же ключ ChaCha20-Poly1305
  защищает **все** сообщения между двумя людьми. Нет эфемерных ключей, ратчета, ротации. Сервер
  ретранслирует/хранит недоставленный шифртекст (офлайн-доставка) — то есть шифртекст перехватываем.
- **Чем грозит:** Поскольку ключ не меняется, если приватный ключ хоть раз получен (изъятие телефона,
  бэкап, вредонос) — расшифровывается **вся** переписка человека, включая старые сообщения, снятые ранее
  с сервера. Signal/WhatsApp постоянно меняют ключи, поэтому вчерашние сообщения остаются закрыты, даже
  если утёк сегодняшний ключ. Все прямые конкуренты (Signal, Threema, Session, даже секретные чаты
  Telegram) имеют forward secrecy — это единственная крипто-фича рынка, которой у Orpheus нет.
- **Как чинить:** Перейти на эфемерные ключи per-session/per-message (Double Ratchet или хотя бы
  ephemeral-static ECDH со свежим эфемерным ключом на сообщение) и периодически ротировать ключ-личность.
- **Трудозатраты:** L

**[SEC-7 / DB-2 / OPS-3] Android Auto Backup включён — незашифрованная БД может уехать с устройства в облако** — `android/app/src/main/AndroidManifest.xml:37`
- **Что не так:** У `<application>` нет `android:allowBackup="false"` и нет `dataExtractionRules`/
  `fullBackupContent`, поэтому `allowBackup` по умолчанию `true`, а каталог `databases` включается в
  авто-бэкап. В связке с SEC-1 это резервируется **незашифрованная** БД.
- **Чем грозит:** При включённом бэкапе Android может автоматически скопировать приватные файлы (в т.ч.
  открытую БД с перепиской, контактами и заметками) в Google-облако пользователя, а на многих устройствах
  `adb backup` вытащит их по USB. Для privacy-продукта это значит, что конфиденциальные разговоры тихо
  покидают телефон и оседают на стороннем сервере — и panic-wipe на телефоне облачную копию не сотрёт.
- **Как чинить:** Поставить `android:allowBackup="false"` (проще всего) либо задать
  `dataExtractionRules`/`fullBackupContent`, явно исключающие `databases` и secure-storage. В связке с SEC-1.
- **Трудозатраты:** S

**[DB-3] Duress-режим — лишь фильтр на уровне запросов поверх одной незашифрованной БД; реальные данные остаются на диске** — `lib/services/database_service.dart:268`
- **Что не так:** Duress реализован тем, что read-методы возвращают пусто при `_isDuressMode` (напр.
  `getContacts:188`, `getMessagesForContact:298`). Все реальные контакты и сообщения остаются в том же
  единственном `orpheus.db`; отдельного скрытого зашифрованного хранилища нет. Хуже: `addMessage`
  (267-274) **без** duress-guard и по собственному комментарию продолжает сохранять входящие в duress —
  то есть реальный (скрытый) набор данных копится, пока телефон у принуждающего.
- **Чем грозит:** Duress-режим призван защитить того, кого заставили разблокировать телефон: принуждающий
  должен увидеть пустое невинное приложение. Но раз реальные данные лишь спрятаны кодом и физически лежат
  в открытом файле, любой, кто смотрит хранилище напрямую (форензика, рут, или тот же авто-бэкап из DB-2),
  видит всю настоящую историю. Фича даёт ложное чувство безопасности ровно в том сценарии, ради которого
  создана.
- **Как чинить:** Реальное разделение: отдельная зашифрованная БД/набор ключей для duress-профиля, чтобы
  реальные данные были криптографически недоступны в duress; перестать писать реальные входящие в
  хранилище, доступное принуждающему. Зависит от SEC-1.
- **Трудозатраты:** L

**[LOGIC-1] Дедуп входящих тихо теряет реальные сообщения, отправленные в пределах 5 секунд** — `lib/services/incoming_message_handler.dart:283`
- **Что не так:** Дедуп идёт по отправителю + времени приёма: `_lastChatTimestampBySender[senderKey]`
  ставится в `_nowMs()`, и любое следующее «chat» от того же отправителя в пределах `_chatDedupeWindowMs`
  (5000 мс) отбрасывается *до* расшифровки/сохранения. Код не различает настоящий дубль (двойная доставка
  WS+офлайн) и просто разные сообщения. Любые два разных сообщения от одного контакта с интервалом <5 с →
  второе теряется.
- **Чем грозит:** В живой переписке люди шлют по 2-3 коротких строки подряд («привет» / «ты тут?»). При
  этом правиле приложение **молча выбрасывает** каждое сообщение, пришедшее в течение 5 секунд после
  предыдущего от того же человека, и пользователю ничего не сообщает. Тихая потеря входящих — один из
  худших багов для мессенджера.
- **Как чинить:** Дедуп по идентичности сообщения, а не по грубому окну: отправитель кладёт стабильный
  `message_id` (или хэш содержимого+времени) внутрь зашифрованного конверта; пропускать только если такой
  id уже сохранён. Убрать 5-секундное окно по отправителю.
- **Трудозатраты:** M

**[LOGIC-2] «Удалить у обоих» никогда не удаляет у получателя (рассинхрон меток времени)** — `lib/services/incoming_message_handler.dart:267`
- **Что не так:** Отправитель шлёт `sendDeleteForBoth(peer, [message.timestamp])` со **своим** временем
  отправки. Получатель сохранил это сообщение с временем **приёма** (`DateTime.now()`,
  `incoming_message_handler.dart:290-295`). Затем `deleteMessagesByTimestamps` матчит по `timestamp IN
  (...)` — время отправки и приёма различаются из-за задержки/расхождения часов, поэтому у получателя
  совпадает 0 строк. Сообщение удаляется только у отправителя.
- **Чем грозит:** Приложение рекламирует «удаление у обеих сторон». Пользователь жмёт, видит, что
  сообщение исчезло у него, и уверен, что оно исчезло и у собеседника — а оно навсегда остаётся в чате
  получателя. Ложное чувство безопасности вокруг фичи, на которую полагаются для чувствительного контента.
- **Как чинить:** Каждому сообщению — общий стабильный идентификатор (UUID от отправителя внутри
  зашифрованной нагрузки), который хранят обе стороны; удалять по id, а не по локальным меткам времени.
- **Трудозатраты:** M

**[PERF-1] Чат грузит всю историю без пагинации и перечитывает весь разговор на каждое сообщение** — `lib/chat_screen.dart:82`
- **Что не так:** `_loadChatHistory()` вызывает `getMessagesForContact()` — `SELECT * ... ORDER BY
  timestamp ASC` без `LIMIT`, возвращая все сообщения. Вызывается при открытии, после каждого
  `_sendMessage` (стр. 127) и на каждое входящее (стр. 63-69). Каждый раз список пересоздаётся, а
  `_buildMessagesList()` ещё и аллоцирует `_chatHistory.reversed.toList()` на каждый rebuild (стр. 344).
- **Чем грозит:** Для того, кто переписывается месяцами, открытие чата становится всё медленнее, а каждое
  новое сообщение заставляет приложение перечитывать весь разговор с диска и перерисовывать список. На
  слабых Android это заметный подтормаживание при каждом входящем и лишний расход батареи — страдают самые
  активные пользователи.
- **Как чинить:** Грузить только последнюю страницу (`LIMIT 50`) и подгружать старое по скроллу — как уже
  сделано в `room_chat_screen.dart`. Для входящих — дописывать одно новое сообщение в память, а не
  перезапрашивать всё. Кэшировать перевёрнутый список.
- **Трудозатраты:** M

**[ARCH-1] Циклическая зависимость двух ядровых сервисов AuthService ↔ DatabaseService** — `lib/services/database_service.dart:19`
- **Что не так:** `database_service.dart` импортирует `auth_service.dart` (стр. 8) и читает
  `AuthService.instance.isDuressMode` (стр. 19), а `auth_service.dart` импортирует `database_service.dart`

## Что сделано хорошо (не трогать)

- **Крипто-примитивы применены верно:** свежий случайный nonce на каждое сообщение, проверка MAC при
  расшифровке (подделка → исключение), приватные ключи только в Keystore-backed `flutter_secure_storage`,
  тяжёлые операции в изолятах (`compute`) — `crypto_service.dart:174/216`.
- **Никакого ослабления TLS:** везде `wss://`/`https://`, нет `badCertificateCallback`/`onBadCertificate`.
- **SQL полностью параметризован** (whereArgs/`?`-плейсхолдеры, вкл. динамические `IN(...)`) — SQL-инъекций нет.
- **FLAG_SECURE на всё приложение** — скриншоты и запись экрана заблокированы глобально (`MainActivity.kt:39-42`).
- **Sentry настроен вдумчиво:** `sendDefaultPii=false`, фильтр breadcrumbs выбрасывает контент сообщений
  (`main.dart:84-93`); закоммичен только клиентский Firebase-ключ (серверных секретов нет); `.gitignore`
  корректно прячет `.env`/keystore/`key.properties`.
- **Прогрессивная защита от брутфорса** (30с→1м→5м→30м) + опциональный auto-wipe; соль 32 байта из `Random.secure()`.
- **Wipe БД сделан тщательно:** закрытие соединения, удаление файла + `-journal/-wal/-shm`, проверка удаления.
- **Входящие WS/FCM разбираются защитно** (проверки типов, `is! Map`, TTL/dedup); FCM аутентифицирован Firebase.
- **Производительность выше среднего:** горячий запрос чата покрыт индексом; непрочитанные — один батч
  `GROUP BY` (N+1 сознательно избегнут); комнаты пагинируются; таймеры/подписки/контроллеры `dispose()`-ятся.
- **Даты хранятся единообразно** как `INTEGER` epoch — нет дрейфа «строки vs эпохи» и ловушек с float.
- **Сильное ядро тестов:** настоящий E2E encrypt→decrypt на двух инстансах; тщательное покрытие PIN/duress/
  wipe/lockout; жест panic с инъектируемыми часами; маршрутизация входящих звонков (dedup/TTL/ICE-буфер).
- **Документация необычно хорошая** для соло/AI-проекта: `docs/` структурированы, ADR ведутся, язык понятный.
- **Зрелая дизайн-система и полная локализация EN/RU**; экран контактов образцово обрабатывает loading/error/empty.
- **AI-ассистент «Oracle» — законченная фича**, не заглушка: стриминг, markdown, персист истории.

## Конкурентный анализ

**Кого анализировал (веб-поиск):** Signal, Threema, Session, Telegram (Secret Chats), VIPole, eXpress,
Max (VK) как контекст рынка. Источники: signal.org, threema.com, getsession.org, core.telegram.org,
vipole.com, express.ms, профильные обзоры (cyberinsider, cryptographyengineering), Wikipedia (eXpress, Max).

**Позиционирование.** Orpheus — в переполненной категории privacy-мессенджеров. Вход **без телефона/почты**
(идентичность = ключ X25519) на уровне Session/Threema и лучше Signal/Telegram/eXpress/Max. **Стек защиты от
принуждения** (duress-профиль, panic/auto-wipe) реально редок среди всех крупных конкурентов. Но по столпам
доверия рынка — open-source, независимый аудит, forward secrecy, защита метаданных, устойчивость к
блокировкам, размер аудитории — Orpheus **отстаёт от каждого серьёзного конкурента**.

|---|---|---|---|---|---|

**Выводы:**
- **ДОБАВИТЬ:** forward secrecy/ратчет (единственная крипто-фича, которой нет только у нас); UX сверки ключа
  (safety number/QR) против MITM; человекочитаемые имена поверх base64-ключа; опубликованную модель угроз +

- **УБРАТЬ / переосмыслить:** позиционирование AI-«Oracle» как ядровой фичи privacy-мессенджера (отправка
  контента в AI противоречит E2E — сделать явно опциональным, off по умолчанию, изолировать от приватных
  чатов); не дрейфовать в «супер-приложение» а-ля Max; убрать мёртвый инлайн-changelog.

- **ПЕРЕДЕЛАТЬ:** крипто-модель → эфемерный ратчет с forward secrecy (самое высокорычажное); trust-историю →
  аудит + open-source крипто-слоя; доступность/метаданные вокруг единого хоста; снизить зависимость от FCM.

**Наше УТП (честно):** сопротивление принуждению (duress-decoy + panic/auto-wipe) — реальное защитимое
преимущество, которого нет «из коробки» у Signal/Threema/Session/Telegram; регистрация без телефона/почты;
свой self-hosted бэкенд (для небольшого доверенного сообщества с одним подотчётным оператором); блок
скриншотов + блокнот + нативные звонки в одном приложении; независимость от Google/Apple и госпаспорта —
противоположность Max/eXpress. **Оговорка без утешений:** по ядровым осям доверия (open-source, аудит,
forward secrecy, метаданные, аудитория) Orpheus **не сильнее** Signal/Threema/Session. Его угол — узкая
ниша: анонимный, устойчивый-к-принуждению мессенджер с одним оператором для высокорисковых русскоязычных
пользователей, а не «самый безопасный мессенджер» вообще.
