# Задача бэкенда: доставка `license-status` по WebSocket — разжёвано

> **ОБНОВЛЕНО 08.07.2026 — ВАЖНО, прочитать первым:** после разбора кода ветки
> `feature/server-tasks-no-crypto` выяснилось, что это **УЖЕ РЕАЛИЗОВАНО ПРАВИЛЬНО** в
> коммите `8893c61`: `notify_license_activated` (`main.py:2173-2189`) шлёт
> `license-status:active` во все живые сокеты pubkey сразу после `activate-promo`, плюс
> ре-чек по `ping` (`main.py:2633-2637`), тесты зелёные. **Переписывать НЕ надо.** Причина,
> по которой на device-тесте пуш не пришёл — **ветка не задеплоена на прод** (прод бьётся с
> mainline). **Реальное действие: смерджить ветку и передеплоить прод**, затем проверить:
> 1 сокет + `POST /api/activate-promo` → `license-status:active` в этот сокет за ~30с.
> Единственная реальная доработка — паритет платёжного пути (вотчер шлёт мимо
> `notify_license_activated`, теряя Redis-fan-out на мультинодe) — детали в
> `SERVER_FINDINGS_2026-07-08.md`. Текст ниже оставлен как объяснение самого механизма.

## В одном предложении
Когда лицензия у пользователя становится активной (ввёл промокод **или** оплатил),
сервер должен **сразу** отправить сообщение `license-status: active` в его **уже
открытый** WebSocket-сокет — не дожидаясь, пока клиент переподключится.

---

## 1. Что такое `license-status` и зачем он нужен

Клиент (мобильное приложение) после запуска открывает постоянный WebSocket:

```
wss://api.orpheus.click/ws/<pubkey>
```

Сервер по этому сокету присылает клиенту статус лицензии сообщением вида:

```json
{ "type": "license-status", "status": "active" }
```
либо
```json
{ "type": "license-status", "status": "inactive" }
```

Клиент по этому сообщению решает: **пускать пользователя в приложение (active) или
держать на экране "Активация" (inactive)**.

> Формат сообщения бери **тот же самый**, что ты уже шлёшь при подключении (см. §3,
> пункт «что работает») — не надо ничего выдумывать, это ровно то же сообщение.

---

## 2. Как ведёт себя клиент (чтобы был понятен контекст)

- При **каждом** открытии WebSocket клиент ждёт от сервера `license-status`.
- Приложение слушает это сообщение **всё время**, пока сокет открыт, а не только в
  первую секунду. Если в любой момент прилетит `license-status: active` — клиент
  тут же пускает пользователя внутрь.
- То есть тебе на сервере достаточно **отправить сообщение в сокет** — клиент уже
  готов его принять и среагировать. Никакого нового API на клиенте не нужно.

---

## 3. Полный сценарий активации по промокоду (где именно ломается)

Пользователь стоит на экране "Активация", вводит промокод, жмёт "Активировать".

Шаги:

1. Клиент шлёт HTTP-запрос `POST /api/activate-promo` (или как он у тебя называется)
   с промокодом. **Это работает.**
2. Сервер проверяет код, **списывает лицензию**, помечает pubkey как лицензированный,
   возвращает HTTP `200 { "status": "ok" }`. **Это работает** (повторный ввод того же
   кода даёт "уже использована" — значит списание прошло).
3. **ВОТ ЗДЕСЬ ПРОБЕЛ:** сразу после шага 2 сервер **НЕ** отправляет
   `license-status: active` в уже открытый сокет этого pubkey. Клиент по WS ничего не
   получает.
4. `license-status: active` прилетает только когда клиент **переподключит** WebSocket
   (например, после перезапуска приложения). На новом коннекте — приходит корректно.

### Что уже работает (НЕ трогать, это правильно):
- ✅ На **первом** коннекте (до активации) сервер шлёт `license-status: inactive`.
- ✅ На **каждом реконнекте после** активации сервер шлёт `license-status: active`.
  Проверено на устройстве: убил приложение → открыл заново → на свежем коннекте
  пришло `active`, пользователь внутри.

### Что надо доделать (суть задачи):
- ❌ **Отправка `license-status: active` в живой (уже открытый) сокет сразу после
  активации** — сейчас её нет. Это единственное, чего не хватает.

---

## 4. Что именно реализовать (псевдокод)

В обработчике активации лицензии (и промокод, и — в будущем — оплата) после того,
как ты пометил pubkey лицензированным, добавь отправку в его открытые WS-сессии:

```python
# после успешной активации лицензии для pubkey
def on_license_activated(pubkey: str):
    mark_license_active(pubkey)                       # это уже есть

    # НОВОЕ: пушим статус во все живые сокеты этого pubkey
    msg = {"type": "license-status", "status": "active"}   # тот же формат, что на коннекте
    for ws in get_open_connections(pubkey):           # у pubkey может быть НЕСКОЛЬКО сокетов
        ws.send_json(msg)                             # (у клиента 2 изолята: main + push)
```

Важные детали:
- **У одного pubkey может быть несколько открытых сокетов** (у клиента два изолята —
  основной и push-сервис). Отправь сообщение во **все** его открытые соединения.
- Если в момент активации **нет** ни одного открытого сокета — ничего страшного,
  клиент получит `active` на следующем коннекте (это уже работает, §3).
- Формат сообщения — идентичен тому, что ты шлёшь на коннекте. Не переименовывай поля.

---

## 5. Как проверить, что готово (критерии приёмки)

Тест на реальном устройстве или в две вкладки:

1. Аккаунт без лицензии, приложение открыто и висит на экране "Активация"
   (WebSocket открыт, до этого пришёл `license-status: inactive`).
2. Ввести промокод и активировать.
3. **Ожидаемо:** в течение 1-2 секунд, **без перезапуска приложения и без
   переподключения WS**, в сокет прилетает `{"type":"license-status","status":"active"}`,
   и приложение само пускает пользователя внутрь.
4. Сейчас (до фикса) этого не происходит — приложение заходит только по HTTP-ответу
   (клиентский обходной путь) или после ручного перезапуска.

Проверить можно по серверным логам: после `activate-promo` должна быть запись об
отправке `license-status: active` в сокет(ы) этого pubkey.

---

## 6. Почему это важно (особенно для будущей оплаты)

- Для **промокода** клиент сейчас прикрыт: он заходит по HTTP-ответу `200 ok`, не
  дожидаясь WS. Так что промокод и без этого фикса работает (проверено на устройстве).
- Но для **оплаты (USDT)** — задача №1 — этот пуш **обязателен**. При оплате нет
  кнопки "Активировать" и нет мгновенного HTTP-ответа "лицензия выдана": пользователь
  оплачивает в своём криптокошельке, а сервер узнаёт об оплате **асинхронно** (вотчер
  ловит транзакцию). В этот момент приложение просто **ждёт по WebSocket**, что
  придёт `license-status: active`. Если сервер не пушит статус в живой сокет —
  пользователь оплатил, но приложение не пустит его, пока он не перезапустит апп.
- Поэтому правильно сделать этот пуш сейчас — он нужен и для промокода, и как
  фундамент для оплаты.

1. Ты уже шлёшь `license-status` на коннекте — это ок, не трогай.
2. Добавь: **сразу после активации лицензии** (промокод/оплата) отправь **то же самое**
   сообщение `{"type":"license-status","status":"active"}` во **все открытые сокеты**
   этого pubkey.
3. Проверка: ввёл промокод → без перезапуска приложение впустило → в логах сервера
   виден пуш статуса в сокет.
