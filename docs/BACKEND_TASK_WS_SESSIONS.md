# ТЗ на серверную часть — управление WS-сессиями и presence

Компонент: **Orpheus Backend** (Python FastAPI, `api.orpheus.click`). Задача
сформулирована по итогам device-теста клиента (2026-07-05). Клиентская сторона
выжата максимум; оставшиеся проблемы — на сервере. Ниже: симптомы → гипотезы
корня → что менять и ЗАЧЕМ → как проверить.

---

## 1. Симптомы (наблюдались на живых телефонах)

**S1. Флап WS у клиента (особенно Samsung/One UI).**
Основной WebSocket клиента молча рвётся каждые ~15-30 секунд: клиент только что
подключился (`WS: Соединение установлено`), а через 10-30с все исходящие уходят в
HTTP-fallback (`WS недоступен`), затем реконнект. В КЛИЕНТСКОМ логе НЕТ причины
обрыва (нет `onDone`, нет ping-таймаута) — сокет закрывается «снаружи». Следствие:
исходящие звонки/сигналы иногда не проходят с первого раза; ответ на звонок на
заблокированном телефоне часто проходит только со 2-го захода.

**S2. Presence-лаг после kill+reconnect клиента.**
Клиент A убили (в т.ч. закрыли постоянное уведомление push-сервиса) → он ушёл
оффлайн. Затем A переоткрыли, его WS поднялся (`Соединение установлено`, лицензия
active) — но у собеседника B он ДОЛГО висит оффлайн, и звонок B→A не проходит,
пока «само не рассосётся» (через десятки секунд).

Обе проблемы — про то, как сервер трекает и чистит WS-сессии одного пользователя.

---

## 2. Контекст клиента (важно для понимания)

Точка входа: `wss://api.orpheus.click/ws/<pubkey_urlencoded>` (публичный ключ
пользователя — идентификатор сессии).

**У одного пользователя (pubkey) МОЖЕТ быть ДВА одновременных WS-подключения:**
- **main-изолят** (UI приложения) — активен, пока приложение живо.
- **push-изолят** (`PushConnectionService`, постоянный foreground-сервис — замена
  FCM после де-гуглизации) — держит доставку при убитом/фоновом приложении.

По задумке push «молчит», пока main жив (координация через heartbeat в самом
клиенте). НО на агрессивных OEM (Samsung One UI) система душит/убивает main-изолят
под локом → push активируется → и тогда ОБА могут подключиться к серверу с ОДНИМ
pubkey примерно одновременно.

**Клиентский keepalive:** клиент шлёт app-level `{"type":"ping"}` каждые 30с и
ожидает ответ; плюс на самом WebSocket выставлен `pingInterval = 10s`. Клиент
считает сессию живой, пока идёт ping-pong.

**Типы сообщений, которые ходят через WS:**
- Presence: `presence-subscribe` / `presence-unsubscribe` (клиент→сервер, список
  pubkeys), `presence-state` (сервер→клиент, снимок статусов после подписки),
  `presence-update` (сервер→клиент push при смене онлайн/оффлайн наблюдаемого).
- Сигналинг звонков (сервер релеит от A к B): `call-offer`, `call-answer`,
  `ice-candidate`, `hang-up`, `call-rejected`, `ice-restart`, `ice-restart-answer`.
  У них есть поле `call_id` — сервер ДОЛЖЕН его сохранять при пересылке (клиент
  дедупит и матчит звонки по нему).
- `license-status`, `ping`/`pong`.

**HTTP-fallback:** для «важных» сигналов (call-offer/answer/ice-candidate/hang-up/
call-rejected) клиент при недоступном WS шлёт их POST-ом. ВАЖНО: HTTP-fallback —
только на ОТПРАВКУ; ВХОДЯЩИЕ сигналы клиент получает ТОЛЬКО по WS. Поэтому если WS
приёмника лежит, звонок ему не собрать (входящие ICE-кандидаты не дойдут).

## 3. Гипотезы корня (проверить на сервере)

**H1 — конфликт двух сессий на один pubkey (наиболее вероятно для S1).**
Если сервер держит «одну сессию на pubkey» и при новом WS-подключении ЗАКРЫВАЕТ
предыдущее — то main-WS и push-WS Samsung ВЫБИВАЮТ ДРУГ ДРУГА по кругу: main
подключился → push подключается → сервер закрыл main → main реконнектится → закрыл
push → и т.д. Это точно даёт «молчаливый обрыв каждые 15-30с» без причины на
клиенте. На Pixel push молчит → конфликта нет → там стабильно.

**H2 — сервер не детектит мёртвую сессию (для S2 и части S1).**
При обрыве TCP без штатного WS-close (телефон ушёл в сон/сеть моргнула) сервер
может держать сессию «онлайн» до долгого TCP-таймаута. Тогда: presence висит
онлайн, входящие сигналы роутятся в мёртвый сокет, а новое подключение того же
pubkey создаёт конфликт (см. H1). Нужен активный keepalive на серверной стороне.

**H3 — presence не ребродкастится при (ре)подключении (для S2).**
Когда наблюдаемый пир A переподключается, сервер обязан ЗАПушить подписчикам
`presence-update {A, online:true}`. Если этого нет — B узнает об онлайне A только
когда B САМ переподключится и заново подпишется (`presence-state`). Отсюда лаг.

## 4. Что менять на сервере (с обоснованием)

### 4.1 Политика нескольких подключений на pubkey  [под S1/H1]
Выбрать ОДНО из:
- **Вариант A (предпочтительно): разрешить несколько живых подключений на pubkey**
  и доставлять входящие сообщения/сигналы ВО ВСЕ живые подключения этого pubkey
  (fan-out). Тогда main и push не мешают друг другу; клиент сам дедупит (по
  `call_id` / message_id). Плюс это устойчиво к любому числу устройств пользователя.
- **Вариант B: аккуратный takeover** — при новом подключении помечать его как
  «живое», старое НЕ закрывать мгновенно, а дать ему протухнуть по ping-таймауту
  (§4.2). Главное — исключить «пинг-понг закрытий» (не закрывать соединение,
  которое активно шлёт ping).

Обоснование: текущее поведение (мгновенное закрытие старого при новом коннекте)
при клиентской архитектуре main+push даёт постоянный флап. Fan-out (A) убирает
проблему в корне и проще в рассуждении.

### 4.2 Активный keepalive + быстрая чистка мёртвых сессий  [под S1/S2/H2]
- Сервер шлёт WS **ping** каждые ~15-20с и требует **pong**; нет pong за ~2
  интервала → сессия считается мёртвой, закрывается и чистится. (Либо отвечать на
  клиентский `{"type":"ping"}` и трекать его периодичность — 30с; молчание >~45-60с
  = мёртв.)
- Включить TCP keepalive на сокете.
- При закрытии/протухании сессии: снять её из реестра pubkey→сессии, и если у
  pubkey не осталось живых сессий — пометить оффлайн + ребродкаст (§4.3).

Обоснование: без активного keepalive мёртвая сессия висит «онлайн», ломает presence
и роутинг, и провоцирует конфликт с новым подключением.

### 4.3 Presence: ребродкаст при (ре)подключении и отключении  [под S2/H3]
- При появлении ПЕРВОГО живого подключения pubkey A → статус `online`, ЗАПушить
  `presence-update {pubkey:A, online:true}` ВСЕМ, кто подписан на A.
- При исчезновении ПОСЛЕДНЕГО живого подключения A (штатный close или протухание) →
  `online:false`, ЗАПушить подписчикам.
- `presence-subscribe` от B на список pubkeys → сразу ответить `presence-state`
  (актуальный снимок), и далее слать `presence-update` при изменениях.
- Идемпотентность: повторные подключения/подписки не должны слать дублирующие/
  противоречивые апдейты.

Обоснование: клиент реагирует на `presence-update`/`presence-state`, но инициатива
за сервером; без ребродкаста при реконнекте пира клиент не узнаёт об онлайне.

### 4.4 Роутинг сигналов только в живую сессию  [под S1/S2]
- `call-offer/answer/ice-candidate/hang-up/call-rejected/ice-restart*` для pubkey B
  доставлять во ВСЕ живые сессии B (fan-out, §4.1-A) — гарантированно попадёт в ту,
  что реально жива. Сохранять поле `call_id` при пересылке.
- НЕ терять сигнал, если единственная известная сессия оказалась мёртвой —
  предпочтительно детектить мёртвую раньше (§4.2). (Опционально: короткая очередь
  недоставленных сигналов на N секунд — но это второстепенно, приоритет — не
  держать мёртвые сессии.)

## 5. Как проверить (сценарии приёмки)

1. **Два подключения на pubkey не флапают:** открыть main + эмулировать push с тем
   же pubkey → оба живут минутами, входящие приходят в оба, никто никого не
   закрывает. (Логи сервера: нет пар close↔reconnect каждые 15-30с.)
2. **Мёртвая сессия чистится быстро:** резко оборвать сокет (kill клиента без
   close) → в течение ~30-45с сервер помечает pubkey оффлайн и шлёт подписчикам
   `presence-update {online:false}`.
3. **Presence при реконнекте:** B подписан на A; A переподключается → B получает
   `presence-update {A, online:true}` в течение пары секунд (без переподписки B).
4. **Звонок в только что переподключённого:** после сценария S2, B→A звонок
   доходит сразу (входящие ICE долетают до A по WS).
5. Регресс: обычные звонки/переписка на стабильной сети продолжают работать.

---

## 6. Просьба к серверному разработчику
Прежде чем менять — подтвердить/опровергнуть H1: **закрывает ли сервер сейчас
предыдущее WS-подключение при новом коннекте того же pubkey?** Это ключевая
развилка. Дальше — по §4. После оценки решим порядок и объём. Клиентскую сторону
(типы сообщений, keepalive, HTTP-fallback) менять не требуется — контракт описан
в §2.
