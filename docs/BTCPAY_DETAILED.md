# BTCPay Server — максимально подробно (под Orpheus)

Проработка Этапа 2 оплаты (макс. анонимность). Активный трек сейчас — TON-бот
(`TON_BOT_SPEC.md`); BTCPay — следующий шаг. Факты сверены с офиц. доками
(docs.btcpayserver.org, github.com/btcpayserver) на 2024-2025; версионно-зависимое
помечено ⚠.

---

## 1. Что это и почему подходит
BTCPay Server — open-source, **self-hosted**, **non-custodial** платёжный процессор.
Без комиссий процессора, без KYC, без третьей стороны, **first-class Tor**. Валидируешь
платежи на СВОЁМ ноде — данные платежей остаются между тобой и клиентом. Это его
главное преимущество по нашей концепции анонимности/суверенности.

---

## 2. Какие монеты (ВАЖНАЯ ПОПРАВКА к нашим планам)
- **Нативно (из коробки):** только **Bitcoin on-chain + Lightning** (LND / Core
  Lightning). Всё остальное — плагины поверх.
- **Monero (XMR):** есть, но **community-плагином, не в ядре**. В BTCPay 2.0 XMR убрали
  из ядра; с 2.1.0 — через `btcpayserver-monero-plugin` (финансирование Magic Grants +
  Monero CCS). Статус: активно поддерживается, но сам себя описывает как выходящий из
  **proof-of-concept/MVP** — фактически **beta**. Требует `monerod` + `monero-wallet-rpc`.
  ⚠ Перед боевым использованием проверить стабильность релиза и мин. версию BTCPay.
- **Litecoin, ETH/ERC-20, Liquid-активы (в т.ч. L-USDt / Tether на Liquid):** через
  плагины/сообщество.
- **❌ USDT-TRC20 (Tron) — НЕ поддерживается (подтверждено).** Единственный USDT у
  BTCPay — L-USDt на Liquid (другая сеть), не Tron.
- **❌ TON (Toncoin) — НЕ поддерживается (подтверждено).** Ни в ядре, ни плагином.

**Вывод для нас:** BTCPay = **BTC + Lightning + Monero**. Наш TON-бот (Этап 1) и BTCPay
(Этап 2) — это **разные платёжные рельсы**, BTCPay не заменяет TON. Для USDT-TRC20/TON
BTCPay не годится — см. §7.

## 3. Non-custodial: асимметрия BTC vs XMR (ключевое)
- **Bitcoin — полностью non-custodial, watch-only.** Вставляешь **xpub** (расширенный
  ПУБЛИЧНЫЙ ключ); BTCPay генерит адрес на инвойс, **приватный ключ на сервер не
  попадает**. Даже полностью скомпрометированный BTCPay **не может увести средства** —
  только смотреть. Поддерживаются xpub с аппаратных кошельков.
- **Monero — нужен HOT-кошелёк (реальный риск).** У Monero нет watch-only, умеющего то,
  что нужно BTCPay, поэтому плагин крутит `monero-wallet-rpc` — **разлоченный кошелёк
  со spend-ключами на сервере**. Взлом сервера = **XMR под угрозой** (в отличие от BTC).
  ⚠ Плагин делит ОДИН Monero-кошелёк на ВСЕ магазины инстанса — «используй, только если
  не шаришь инстанс». Это главная асимметрия: макс. анонимность (XMR) = макс. hot-wallet
  риск. Митигация: регулярно сметать XMR в холодное хранилище, изолировать инстанс.

## 4. Требования к самохостингу
- **Развёртывание:** офиц. **Docker** (`btcpayserver-docker`, `btcpay-setup.sh`) —
  стандарт. Есть managed (LunaNode, Voltage) — но это снижает приватность самохостинга.
- **Bitcoin-нод, pruned vs full:**
  - **Pruned (рекомендуется для приёма):** ~**10-15 ГБ** цепи; комфортно на **80 ГБ
    SSD/NVMe VPS**, минимум **2 CPU / 4 ГБ RAM**, комфорт **4 CPU / 8 ГБ**. Первичная
    синхронизация ~6-24 ч. VPS ~€10-30/мес.
  - **Full archival:** **500+ ГБ**. Нужен редко.
- **Monero добавляет нагрузку:** `monerod` + `monero-wallet-rpc`; блокчейн Monero
  большой и быстро растёт (**200+ ГБ**, ⚠ уточнить актуальный размер). Доки прямо
  предупреждают: Monero «не рекомендуется для неопытных», может подвесить слабый сервер.
  Включается `BTCPAYGEN_CRYPTO2="xmr"` + `btcpay-setup.sh -i`.
- **Lightning:** LND или Core Lightning; нужна ликвидность каналов + бэкапы состояния.

## 5. Интеграция (Greenfield API — то, что делает наш FastAPI-бэкенд)
- **Создать инвойс:** `POST /api/v1/stores/{storeId}/invoices`, заголовок
  `Authorization: token <API_KEY>`.
  - Тело: `amount`, `currency` (фиат или крипта), **`metadata`** (свободное; сюда
    кладём **`orderId` + pubkey / session_token** — так инвойс привязывается к
    пользователю), `checkout` (`expirationMinutes`, `speedPolicy`, `paymentMethods`,
    `paymentTolerance`, `redirectURL`).
  - Полная схема на инстансе: `/docs` (Swagger) и `/swagger/v1/swagger.json`.
- **Состояния инвойса:** `New → Processing → Settled`, терминальные `Expired`, `Invalid`.
  Модификаторы: `paidPartial`, `paidLate`, `paidOver`, `marked`.
  (New = не оплачен, таймер идёт; Processing = оплачен, ждём подтверждений; Settled =
  оплачен + достаточно подтверждений; Expired = не оплачен, таймер вышел; Invalid =
  оплачен, но не подтвердился вовремя.)
- **Вебхуки (события):** `InvoiceCreated`, `InvoiceReceivedPayment`, `InvoiceProcessing`,
  `InvoicePaymentSettled`, `InvoiceExpired`, `InvoiceInvalid`, `InvoiceSettled`.
  Регистрация: `POST /api/v1/stores/{storeId}/webhooks`.
  - **Лицензию активируем на `InvoiceSettled`** (оплачено И подтверждено). НЕ на
    `Processing` для on-chain (риск неподтверждённой/replaced-tx). Для Lightning
    Processing≈Settled (мгновенно).
  - **Проверка подписи:** заголовок **`BTCPay-Sig`** = **HMAC-SHA256** сырого тела
    запроса на **секрете вебхука**; сравнивать **timing-safe**. В payload `invoiceId` →
    коррелируем с `orderId`/pubkey.

## 6. Курс и цены
- **Rate-провайдер + фиксация:** при создании инвойса BTCPay конвертит фиат→крипту по
  настраиваемому источнику курса и **фиксирует сумму на срок инвойса** (защита от
  волатильности).
- **Срок инвойса по умолчанию — 15 минут** (офиц.), настраивается (`expirationMinutes`).
  ⚠ Часть сторонних гайдов пишет 5 мин — ставить явно. Медленные сети + короткий срок =
  частый `paidLate`.
- **Недоплата/переплата:** недоплата → `New(paidPartial)`, по таймауту `Expired(paidPartial)`
  (частая причина — биржа удержала комиссию вывода); можно задать `paymentTolerance` (%).
  Переплата → `Settled(paidOver)`, излишек — возврат вручную. Поздняя оплата → `paidLate`.

## 7. Приватность
- **Tor:** first-class. Docker включает **hidden service (.onion) по умолчанию**
  (`opt-add-tor`). Можно clearnet / onion / оба.
- **Нет посредника:** self-hosted = **никто третий не видит транзакции**; валидация на
  своём ноде.
- **Ограничения:** on-chain **BTC псевдонимен и прослеживаем** (per-invoice адреса
  помогают, но chain-analysis работает); **Lightning** лучше, но не идеал; **XMR** —
  сильная анонимность (за счёт hot-wallet риска). ⚠ Запросы курса к биржам утекают
  метаданными — гнать через Tor/SOCKS.

## 8. Альтернативы для USDT-TRC20 / TON (BTCPay не умеет)
- **NOWPayments** — есть **USDT-TRC20** + TON + 350 монет; **non-custodial по умолчанию**
  (форвардит на твой кошелёк). НО это **сторонний процессор** — видит транзакции, для
  онбординга нужен аккаунт/API → эрозия анонимности vs BTCPay.
- **CoinPayments** — очень широкое покрытие, но **custodial по умолчанию + KYB** —
  худший фит по анонимности.
- **Общий трейд-офф:** любой хостед-шлюз с Tron/TON = снова посредник и обычно KYC.
  Правильный паттерн: **BTCPay self-hosted для BTC+Lightning+XMR**, а USDT-TRC20/TON —
  ОТДЕЛЬНОЙ рельсой через non-custodial NOWPayments, только если спрос оправдывает
  потерю приватности. (У нас TON уже закрыт ботом Этапа 1 — см. `TON_BOT_SPEC.md`.)

## 9. Крайние случаи / эксплуатация (продажа цифровых лицензий)
- **Активировать на `InvoiceSettled`** (не Processing) для on-chain; Lightning — мгновенно.
- **Время подтверждений (UX):** BTC ~10 мин/блок (1 conf ≈10 мин, 6 conf ≈1 ч); LTC
  ~2.5 мин; **XMR ~2 мин/блок, часто ~10 подтверждений ≈20 мин**; **Lightning — мгновенно**.
  ⚠ Против 15-мин срока инвойса медленные сети дают `paidLate` — ставить `expirationMinutes`
  щедро на метод или вести на Lightning.
- **SpeedPolicy (сколько подтверждений до Settled):** `HighSpeed/MediumSpeed/
  LowMediumSpeed/LowSpeed` (условно 0/1/2/6 conf). ⚠ Дефолт плохо задокументирован —
  **ставить `speedPolicy` явно**; для on-chain НЕ брать High (0-conf, риск double-spend).
- **Hot-wallet XMR:** сметать в холод регулярно, изолировать инстанс.
- **Возвраты:** через **Pull Payments** — клиент сам заявляет адрес, ты
  подтверждаешь/подписываешь/бродкастишь. **Ручные, не авто**; для BTC-возврата нужен
  spending-кошелёк (watch-only xpub возврат сделать не может). Нужна политика для
  paidPartial/paidOver/paidLate.
- **Обслуживание:** синхронизация нода, бэкапы Lightning-каналов (потеря состояния =
  потеря средств), обновления, рост диска (особенно Monero), безопасный бэкап
  seed/xpub, БД и секретов вебхуков. Даунтайм = клиент не может платить.

## 10. Рекомендация для Orpheus (Этап 2)
- Держать **BTCPay self-hosted** для **BTC (xpub, non-custodial) + Lightning
  (мгновенно) + Monero (макс. анонимность)**.
- **XMR** = наш козырь по анонимности, но помнить про hot-wallet: изолированный
  инстанс, регулярный sweep в холод.
- Активировать лицензию строго на `InvoiceSettled`, проверять `BTCPay-Sig` (HMAC),
  `metadata.orderId`+pubkey для привязки. Тот же контракт «оплата → активация лицензии»,
  что и у TON-бота.
- USDT-TRC20/TON держать на TON-боте (Этап 1) / при необходимости NOWPayments —
  отдельной рельсой, осознавая потерю приватности.
- ⚠ Перед боем перепроверить: зрелость Monero-плагина, размер блокчейна Monero,
  дефолт `speedPolicy` и `expirationMinutes` (ставить явно).
