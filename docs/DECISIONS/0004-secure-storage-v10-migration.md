# ADR 0004: Апгрейд flutter_secure_storage 9 → 10 и переход на современные шифры

## Статус
Принято (2026-07-02). Требует проверки на устройствах перед релизом существующим
пользователям — см. `docs/SECURE_STORAGE_V10_CHECKLIST.md`.

## Контекст
`flutter_secure_storage` хранит НЕвосстановимые секреты приложения: приватный
X25519-ключ (личность), хэши PIN/duress/wipe + соль, ключ шифрования БД (SQLCipher).
Потеря любого = необратимая потеря аккаунта и нечитаемая переписка.

Приложение использовало `const FlutterSecureStorage()` без опций — то есть дефолтный
Android-бэкенд v9 (ключ RSA/ECB/PKCS1, данные AES/CBC), данные в обычных SharedPreferences.

Апгрейд на v10 нёс скрытый риск: v10 сменил ДЕФОЛТНЫЕ шифры (ключ → RSA-OAEP, данные →
AES-GCM) и по умолчанию при сбое авто-миграции из старого формата **безвозвратно стирает**
хранилище (`resetOnError: true`, `migrateWithBackup: false`). Для приложения без
серверного бэкапа ключей это документированная потеря данных и краши на старте
(juliansteenbakker/flutter_secure_storage issues #1043, #1079). Наивный апгрейд «в лоб»
затирал/ронял аккаунты существующих пользователей.

## Рассмотренные варианты
1. **Остаться на v9.** Защитимо (v9 не сломан, security-фикса в v10 нет), но откладывает
   неизбежное (старые шифры удалены в v11).
2. **Мост на старых шифрах v9** (`RSA_ECB_PKCS1Padding` + `AES_CBC_PKCS7Padding`,
   `migrateOnAlgorithmChange: false`, `resetOnError: false`). Сохраняет данные v9, но это
   костыль: обновляемся, но выключаем всё новое; старые шифры deprecated и удалены в v11.
3. **Честный переход на современные шифры v10, данные v9 пожертвовать.** Возможно, т.к.
   приложение в закрытой бете и владелец согласовал жертву данными.

## Решение
Выбран вариант 3 — **честный переход на современные шифры v10**.

- Единый источник опций `lib/services/secure_storage_options.dart` (`appSecureStorage`),
  используется во ВСЕХ местах (CryptoService, AuthService, DatabaseService). Опции: шифры
  v10 по умолчанию (RSA-OAEP + AES-GCM), `resetOnError: false`, `migrateOnAlgorithmChange: false`.
- Авто-миграцию v10 из формата v9 НЕ используем (ненадёжна — краши/частичное стирание).
  Вместо неё `ensureSecureStorageMigrated()` — детерминированный одноразовый `deleteAll()`
  при первом запуске (флаг `secure_storage_v10_reset_done` в SharedPreferences),
  выполняется в `main.dart` ДО первого чтения ключей. `deleteAll` не расшифровывает данные,
  поэтому старый формат стирается без риска краша.

## Последствия
- **Существующие бета-пользователи теряют старый аккаунт** (личность/переписку) и создают
  новый после обновления. Согласовано (закрытая бета).
- Новые пользователи — сразу на современных, v11-совместимых шифрах. Никакого deprecated-кода.
- `minSdk` без изменений (v10 требует 23, у нас 23). API read/write/delete не менялся.
- Перед релизом существующим пользователям ОБЯЗАТЕЛЕН девайс-чеклист (главный риск —
  старт без краша после обновления v9→v10 поверх, на разных OEM).
- Будущее: настоящий смысл `resetOnError: false` раскроется, когда появится экспорт/импорт
  recovery-ключа; тогда можно будет пережить и сбой Keystore без потери аккаунта.
