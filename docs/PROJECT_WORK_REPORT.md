# Orpheus Client — полный отчёт по работе (с начала, после аудита)

Для коллеги-кодера. Что делали, как и зачем — от старта работ (2026-07-01) до
device-теста (2026-07-05). Ветка `wl/dev` (независимый трек, актуальный Flutter;
upstream `master` не трогали). `flutter analyze` — 0 ошибок, `flutter test` — 353
проходят. Полная хронология — в `AI_WORKLOG.md`, пользовательский changelog — в
`CHANGELOG.md`, аудит — в `AUDIT_REPORT.md`.

Отправная точка: **внешний аудит** кодовой базы (коды вида SEC-*, LOGIC-*, UI-*,
DB-*, PERF-*, ARCH-*, DEP-*, PROD-*, OPS-*, QUAL-*). Работали фазами: сначала
закрыли аудит, потом де-гуглизация, апгрейд зависимостей, рефакторы, и в конце —
живой device-тест на двух телефонах.

## ФАЗА 1 — закрытие аудита (07-01 … 07-02)

### Безопасность (SEC/LOGIC)
- **SEC-1 `992ffde`:** шифрование локальной БД at-rest через **SQLCipher**.
- **SEC-2 `d424fc5`:** телеметрия сделана **opt-in**; перестали выгружать
  метаданные контактов/устройства наружу.
- **SEC-8/LOGIC-5 `c1b0e41`:** PIN/duress/wipe хешируются реальным **Argon2id**
  (было слабое хеширование), `verifyPin` стал async (через `compute()`).
- **SEC-5/LOGIC-7/ARCH-7 `c839a5b`:** panic-wipe сделан исчерпывающим,
  best-effort, реально затирает живой ключ (раньше мог оставлять данные).
- **LOGIC-9 `87484a3`:** устранена гонка реконнекта WebSocket, создававшая два
  живых сокета одновременно.
- **LOGIC-6 `0957db8`:** блокировка брутфорса PIN — на **монотонных часах**, а не
  wall-clock (защита от перевода времени).
- **QUAL-1/OPS-6 `343f6c3`:** перестали писать security-события в logcat в release.
- **LOGIC-8 `b3c7365` + follow-ups:** офлайн-лицензия корректно обрабатывается;
  сервер — источник истины (grace-период потом убран по решению, `e300210`).

### Сообщения / БД (LOGIC/DB)
- **LOGIC-1/2, DB-4 `ae1c384`:** стабильный `message_id` → дедуп + «удалить у обоих».
- **LOGIC-3/4, UI-8 `dc1ff3f`:** реальный статус исходящих (раньше всегда
  «доставлено»).
- **DB-6 `0a792b7`:** неизвестные отправители авто-добавляются в контакты, иначе
  их сообщения не видны.

### Производительность (PERF)
- **PERF-1/3 `d152234` `ca2b789`:** инкрементальная/пагинированная загрузка чата +
  debounce обновления контактов.
- **PERF-2 `dda991f`:** индекс `messages.timestamp` для очистки истории.
- **PERF-4 `2b8810b`:** дедуп одновременных in-flight запросов бэджей.

### Продукт / UX (PROD/UI)
- **PROD-1/ARCH-5/SEC-9 `0d6031b`:** удалён **Desktop Link** (мёртвый код +
  небезопасный протокол паринга; вернём безопасно после клиента+сервера).
- **PROD-2/QUAL-3 `2708552`:** тумблер биометрии сделан настоящим (раньше не
  подтверждал биометрией при включении).
- **PROD-3/4/5/6:** поиск по контактам, время последнего сообщения, меню «…» в
  чате вместо one-tap-очистки, онбординг при первом запуске.
- **UI-3/4/5/7/8/9:** guard/loading на создании аккаунта, состояния QR-камеры,
  валидация контакта, отладочный лог-оверлей спрятан в release.

### Инфраструктура (OPS)
- **OPS-1 `11e4d2b`:** подпись release реальным keystore (`key.properties`,
  fallback на debug).
- **OPS-2 `5dcd146`:** постоянный `applicationId = click.orpheus.app`.

## ФАЗА 2 — де-гуглизация (07-02)  [ключевое направление проекта]

Цель владельца — полная независимость клиента от сервисов Google (privacy).
- **Фаза 1 `40a0d7e` `5a26036`:** шрифты Inter забандлены локально (без
  `google_fonts` runtime-фетча); выкинуты Firebase Analytics; Sentry за opt-in
  флагом; убран сторонний ip-api.
- **Фаза 2/3 `e40383f` `953f...`:** удалены **ML Kit** (QR без него) и **FCM**;
  вместо FCM — постоянный foreground-сервис (`specialUse`) `PushConnectionService`
  с WebSocket в отдельном изоляте, доставляющий пуши/звонки при убитом приложении.
- Итог: **ноль gms/firebase/mlkit** классов, работает на GrapheneOS без GMS
  (подтверждено). Это архитектурно важно: у клиента ДВА WS — main-изолят + push-
  изолят (см. серверное ТЗ, это источник части проблем с сессиями).
- `12316f2`: нативный микрофонный foreground-сервис — звук в звонке переживает
  сворачивание (Android 14 правила FGS).

## ФАЗА 5 — живой device-тест (07-03 … 07-05)  [основной объём отладки]

Прогон на Samsung SM-S938B (One UI) + Pixel 7 Pro (GrapheneOS). Фокус — звонки,
приватность и безопасность на локскрине. Найдено и закрыто много того, что
всплывает только на железе.

### 5.1 Персистентность аккаунта (главный блокер) `b0492d8`
- secure_storage v10 на strict KeyMint (Titan M2 / Knox): RSA-OAEP/MGF1-SHA1
  отклонялся при рестарте → аккаунт слетал. Фикс: `RSA_ECB_PKCS1Padding` +
  `AES_GCM_NoPadding` + `resetOnError`. Подтверждён на устройстве.

### 5.2 Звонки: сигналинг/дедуп
- **Дубль SDP-answer** `b284a06`: гейт `handleAnswer` по `HaveLocalOffer` + catch.
- **Чужой отбой** `66ed8ef`: `hang-up/call-rejected` завершал по `call_id` (был
  без проверки → устаревший reject убивал текущий звонок).
- **Двойной экран** `08ed016`/`24aa632` (найдено multi-agent аудитом): дедуп
  опирался на `isCallActive`, выставляемый ПОЗДНО (в `initState`, кадром позже
  push) → синхронный CLAIM по `call_id` (`_claimCallNavigation`); убран
  редундантный fullScreenIntent-аффорданс.
- **call_id не освобождался** `b1ca32d`: `clear()` был мёртвым кодом → добавлен на
  ВСЕ пути завершения.
- **Перезвон в 30с** `b6cca82`: `generateFallbackCallId` давал одинаковый id внутри
  30с-окна → `generateUniqueCallId` (микросекунды) для исходящего.

### 5.3 Приватность имени на локскрине `fe491b3` `988e5ac` `3dbc92f`
- «Закрытая связь / Secure connection» когда приложение не на переднем плане;
  реализовано через флаг `app_in_foreground` в SharedPreferences (читается и в
  push-изоляте, с `prefs.reload()`), + настройка (по умолч. ВЫКЛ).

### 5.4 Безопасность на локе
- **Авто-ответ заблокированного** `51b0780`: fullScreenIntent-уведомление на локе
  само дёргало обработчик → открывался экран с `autoAnswer:true` → трубка сама.
  Фикс: тап уведомления → `autoAnswer:false`.
- **Обход PIN приложения** `e5ecebf` (важно): `LockScreen` был `home`, pushed-
  маршруты перекрывали лок. Фикс — LockScreen оверлеем поверх Navigator.

### 5.5 Answer-over-lock — ответ на звонок на заблокированном (эпопея, ~6 подходов)
Итог — РАБОТАЕТ (~5с после PIN, иногда со 2-го захода из-за флапа WS Samsung):
- Тайминг showWhenLocked: из `initState` (поздно) → в `MainActivity.onStart`
  (плагин переиспользует singleTop-активити, `onCreate` не вызывается) `807cd80`/`ff0a9bd`.
- **Форк плагина** `364f8fd`: в `flutter_callkit_incoming` `onAcceptClick` сам
  звал `requestDismissKeyguard` → PIN устройства. Убрали (см. §6).
- Детерминизм: `isAppInForeground && !requiresUnlock` → под локом всегда CallKit+
  pending → PIN приложения → соединение.
- Пере-обмен: `processPendingCallAfterUnlock` ждёт WS Connected `01533b9`; ICE-
  restart watchdog в CallScreen `d7235fb`/`cd443be` — ранние кандидаты звонящего
  теряются при отложенном ответе, watchdog пере-запрашивает их.

### 5.6 Живучесть WS `7cb71ed` `261db1f`
- WS не поднимался после разблокировки (старт-connect гейтился на `!_isLocked`,
  `_onUnlocked` не поднимал) → добавлен подъём на unlock.
- connect-watchdog 20→8с; `forceReconnectIfStale` на resume (закрыть мёртвый сокет
  + reconnect, если WS не Connected после фона).

## ИЗВЕСТНЫЕ ОГРАНИЧЕНИЯ / ОТКРЫТОЕ (→ серверная часть)

1. **Флап WS у Samsung**: основной WS молча рвётся каждые ~15-30с (нет
   onDone/ping-таймаута в клиентском логе). Из-за этого ответ на локе иногда со
   2-го захода. Клиентски выжато максимум.
2. **Presence-лаг после kill+reconnect**: после убийства+переоткрытия клиента
   собеседник видит его оффлайн + звонок в него не проходит, пока сессия не
   протухнет.
3. Оба — ОДИН узел: управление WS-сессиями на бэкенде. См. отдельное **ТЗ на
   серверную часть** (`BACKEND_TASK_WS_SESSIONS.md`).

## Как поддерживать форк плагина
`third_party/flutter_callkit_incoming` (база 3.1.3) — единственное отличие от
upstream — удалён `dismissKeyguard()` в `onAcceptClick`. Подключён через
`dependency_overrides: path`. `tool/check_callkit_fork.ps1` + `ORPHEUS_FORK.md` —
проверка апдейтов и инструкция переприменения.
