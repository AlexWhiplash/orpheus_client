# flutter_secure_storage 9 → 10: девайс-чеклист (ОБЯЗАТЕЛЬНО перед релизом)

Апгрейд `flutter_secure_storage` `^9.0.0` (9.2.4) → `^10.3.1`.

`flutter_secure_storage` хранит НЕвосстановимые секреты: приватный X25519-ключ
(личность), хэши PIN/duress/wipe + соль, и **ключ шифрования БД (SQLCipher)**.
Потеря любого = катастрофа (без ключа БД вся переписка нечитаема).

## Почему это опасно (контекст решения)

v10 сменил дефолтные Android-шифры (ключ RSA PKCS1 → OAEP, данные AES-CBC → AES-GCM)
и по умолчанию при сбое авто-миграции **безвозвратно стирает** хранилище
(`resetOnError: true`, `migrateWithBackup: false`). Для приложения без серверного
бэкапа ключей это документированная потеря данных
(juliansteenbakker/flutter_secure_storage issues #1043, #1079 — в проде, unrecoverable).

## Что сделано в коде (безопасный мост)

Все места создания `FlutterSecureStorage` (CryptoService, AuthService, DatabaseService)
переведены на единые опции из `lib/services/secure_storage_options.dart`:
- шифры зафиксированы на «старых» v9 (`RSA_ECB_PKCS1Padding` + `AES_CBC_PKCS7Padding`)
  → алгоритм НЕ меняется → миграция v10 НЕ запускается;
- `migrateOnAlgorithmChange: false`, `resetOnError: false` → авто-wipe отключён.

Итог: v10 читает данные v9 в исходном формате, риск потери — нулевой.
**Это ВРЕМЕННЫЙ мост:** старые шифры @Deprecated в v10 и удалены в v11. Полноценный
переход на новые шифры — только после появления экспорта/импорта ключа в приложении,
затем `migrateOnAlgorithmChange: true` + `migrateWithBackup: true` + `resetOnError: false`.

`minSdk` не меняется (v10 требует 23, у нас 23). API read/write/delete/deleteAll
не менялся. Сборка debug/release APK проходит.

## КРИТИЧЕСКИЙ тест — путь обновления v9 → v10 (не uninstall!)

Uninstall стирает и SharedPreferences, и Keystore → миграция НЕ проверяется.
Нужно именно **обновление поверх** на ТОМ ЖЕ устройстве.

- [ ] Собрать APK с ПРЕДЫДУЩЕГО коммита (до этого апгрейда, v9.2.4) и установить на реальное устройство.
- [ ] Создать аккаунт (генерируются X25519-ключи), задать **PIN**, **duress-код**, **wipe-код**.
- [ ] Обменяться 2–3 сообщениями с другим устройством (в БД появляются данные + используется ключ шифрования БД).
- [ ] Перезагрузить приложение, убедиться что всё на месте (базовая точка отсчёта).
- [ ] **Обновить поверх** (НЕ удаляя) на APK с этим апгрейдом (v10.3.1), то же устройство.
- [ ] После обновления проверить по пунктам:
  - [ ] Приложение открывается, личность на месте (тот же публичный ключ / fingerprint на экране «Система»).
  - [ ] **Вход по обычному PIN** работает.
  - [ ] **Duress-код** открывает пустой профиль.
  - [ ] **Существующие сообщения читаются** (значит ключ шифрования БД прочитан корректно).
  - [ ] Отправка/приём нового сообщения работает.
  - [ ] **Wipe-код / panic-wipe** срабатывает (`deleteAll` чистит всё).

## Матрица устройств (Keystore-квирки кластеризуются по OEM)

Пройти критический тест выше минимум на:
- [ ] Pixel / чистый Android (напр. Android 14/15)
- [ ] Samsung (One UI)
- [ ] Xiaomi (MIUI/HyperOS)
- [ ] По возможности ещё один OEM (Oppo/Vivo/Huawei)
- [ ] Разные версии Android (напр. 10, 13, 15)

## Дополнительно

- [ ] **После перезагрузки устройства** (не только приложения): открыть приложение, войти по PIN, прочитать сообщения. (У v10 исторически ломалось «чтение после boot».)
- [ ] **Чистая установка** (без данных v9): создание аккаунта, PIN, сообщения, wipe — всё с нуля работает.
- [ ] Биометрия (если включена): вход по отпечатку после обновления.
- [ ] Прогрессивная блокировка после неверных PIN и авто-wipe по счётчику.
