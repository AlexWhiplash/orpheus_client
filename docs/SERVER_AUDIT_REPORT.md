## Фаза 0 — Разведка и диагностика

**Стек:** Python / FastAPI (монолит), SQLAlchemy Core + `databases` (asyncpg / aiosqlite),
Redis (multi-node WS), Firebase Admin (FCM), `bip_utils`/`tronpy` (оплата TRON), OpenAI (Oracle).

**Масштаб:** ~40 `.py`-файлов. `main.py` **2769** строк и `app/admin_api.py` **1672** — два
«файла-бога»; далее `rooms_api.py` 605, `payments.py` 450, `public_api.py` 325. Тесты — 18 файлов.

**Диагностика (только чтение):**
- `pytest` — **149 тестов зелёные** (~13 c). Честные assert'ы, покрыты рисковые зоны (WS/presence/offline/license-gate/rooms/security).
- `py_compile` изменённых патчем файлов — чисто.
- Сырого SQL со склейкой строк не найдено — запросы параметризованы (SQL-инъекций не видно).
- `requirements.txt` — в кодировке **UTF-16** (мелкий риск для pip-инструментов).
- Firebase/FCM всё ещё активны в коде сервера (~28 упоминаний в `main.py` + закоммиченный ключ) — расхождение с курсом de-Google клиента.

**Учтён прошлый разбор + патч:** WS-сессии (fan-out) — корректно; license-status после
активации — код верный (ветка автодеплоится); rooms delete + support pre-auth — готовы;
оплата — частичная (~30%, выключена флагом); RU-changelog — контент-гэп. Патч
`SERVER_SMALL_FIXES.patch` закрывает утечку `internal_notes`, паритет доставки оплаты и
коллизию `tx_id` (помечено ниже как «✅ закрыто патчем»).

## Сводка

Сервер Orpheus — это монолит на FastAPI (Python): один файл `main.py` на ~2770 строк плюс вынесенные роутеры (`app/*_api.py`), фоновый watcher платежей (`payments.py`), мост Redis для многосерверного режима и интеграции с Firebase (push), TRON (оплата), OpenAI (Oracle). Ядро реального времени (WebSocket-сессии, presence, доставка офлайн-сообщений) написано аккуратно и покрыто тестами — 149 тестов проходят за ~13 секунд. Сквозное шифрование 1:1 (E2E) действительно защищает содержимое личных переписок: сервер их не видит.

Но по периметру картина тревожная. Живой приватный ключ Firebase закоммичен в git и вшит в Docker-образ; идентификация пользователя строится на публичном ключе без доказательства владения (кто угодно с чужим pubkey может выдать себя за него для поддержки, комнат, presence); админ-секрет имеет захардкоженное значение по умолчанию. Денежный модуль (оплата USDT) выключен флагом — и это хорошо, потому что в текущем виде он следит не за той монетой (нативный TRX вместо USDT-TRC20), неатомарно выдаёт лицензии и вообще не покрыт тестами. Из 5 продуктовых задач полностью открыта одна крупная — оплата. Процессная дисциплина (changelog, worklog, git-хуки) сильная, но онбординг-документация расходится с кодом, а CI (автопроверки перед автодеплоем) отсутствует полностью.

## Находки

### 🔴 Критично

**[SEC-1] Приватный ключ сервис-аккаунта Firebase закоммичен в git и вшит в Docker-образ** — `serviceAccountKey.json` (корень репо), `Dockerfile:56` (`COPY serviceAccountKey.json* ./`)
- Что не так: файл в `git ls-files` (в HEAD/истории, не только в рабочей копии), содержит реальный `BEGIN PRIVATE KEY`. `.gitignore` его перечисляет, но на уже отслеживаемый файл это не влияет.
- Чем грозит: любой с доступом к репозиторию, истории или образу владеет полными Google-credentials — может слать push всей базе пользователей и действовать от имени вашего Firebase-проекта. Ключ надо считать скомпрометированным.
- Как чинить: немедленно ротировать ключ в Google Cloud Console; `git rm --cached serviceAccountKey.json` + вычистить историю (git filter-repo/BFG) + force-push и переклонирование у всех; грузить ключ рантайм из secret/env, а не из репо; добавить в `.dockerignore`.
- Трудозатраты: S (ротация) + M (чистка истории)

**[SEC-2] Нет доказательства владения приватным ключом — можно выдать себя за любого по его pubkey** — WS `main.py:2594-2599`, `main.py:900-901`; HTTP `X-Pubkey` в `support_api.py:28-69`, `rooms_api.py:52`
- Что не так: идентичность = строка pubkey в URL/заголовке. `websocket_endpoint` просто делает `accept()` без челленджа-подписи. Pubkey не секретен — это контактный идентификатор, которым пользователи обмениваются.
- Чем грозит: посторонний с чужим pubkey читает всю переписку с поддержкой (хранится в открытом виде, `support_api.py:78`), уничтожает офлайн-очередь жертвы (сообщения удаляются после «доставки» атакующему, `main.py:1615-1619`), пишет в комнаты и в поддержку от её имени (`main.py:2725`), подделывает presence, читает историю комнат (серверный plaintext). E2E-содержимое личных 1:1 при этом остаётся защищённым.
- Как чинить: handshake с proof-of-possession — сервер выдаёт nonce, клиент подписывает его приватным ключом, сервер проверяет и выдаёт сессионный токен вместо доверия сырому заголовку.
- Трудозатраты: L

**[SEC-3] Админ-секрет имеет захардкоженное значение по умолчанию, а JWT-секрет наследует его** — `main.py:65-66`
- Что не так: `ADMIN_SECRET = os.getenv("ADMIN_SECRET", "<дефолт>")`; `ADMIN_JWT_SECRET` падает обратно на него. В отличие от `DATABASE_URL` (который роняет старт, если не задан), тут — тихий откат на публичную константу.
- Чем грозит: если переменная не задана в проде, оба пути админ-авторизации (заголовок и подделываемый JWT) схлопываются в один общедоступный ключ → полный захват админки: выдача лицензий, подмена версии приложения и URL загрузки (supply-chain), выдача бейджей, чтение всех пользователей/логов/поддержки.
- Как чинить: fail-fast на старте, если секреты не заданы или равны дефолту; два независимых высокоэнтропийных секрета (JWT не должен наследовать заголовочный).
- Трудозатраты: S

### 🟠 Высокий

**[SEC-4] Нет ограничения попыток на админ-логине, активации промо и поддержке** — `app/auth_api.py:10-27`, `main.py:2131`, `support_api.py` (весь); лимитер есть только в `oracle_api.py`
- Что не так: `admin_login` без счётчика попыток/локаута/задержки; `/api/activate-promo` без троттлинга (bulk-коды — 8 hex-символов, ~32 бита, `admin_api.py:226`); поддержка допускает спам/флуд логов.
- Чем грозит: онлайн-брутфорс админ-пароля (прямой путь к захвату из SEC-3) и перебор промокодов = потеря выручки.
- Как чинить: rate limiting по IP + аккаунту и временный локаут на логине; троттлинг на activate-promo и POST поддержки; ключ лимитера — по IP, не по подделываемому заголовку.
- Трудозатраты: M

**[SEC-10] `GET /logs` отдаёт серверные логи без всякой авторизации** — `main.py:1971-1988`
- Что не так: в отличие от соседних `/admin/*`, у этого эндпоинта нет ни `x_admin_secret`, ни JWT. Отдаёт HTML с содержимым `logs_table` и принимает `?pubkey=` с фильтром `LIKE`. Плюс `message` вставляется в HTML без экранирования.
- Чем грозит: для «zero-knowledge» мессенджера — утечка метаданных (коннекты, активации лицензий, звонки) с фильтром по ключу конкретного пользователя, доступная любому.
- Как чинить: та же проверка секрета/JWT, что на `/admin/*` (одна строка) + `html.escape()`.
- Трудозатраты: S

**[MONEY-1] Watcher подтверждает оплату по нативному балансу TRX, а не по USDT-TRC20** — `payments.py:195-208` (`get_trx_balance`), `242` (`bal >= expected`), `PAYMENT_BASE_PRICE=10 TRX`
- Что не так: читается нативный TRX-баланс адреса, а продукт — USDT-TRC20. Нет проверки контракта токена, точной суммы, отправителя и tx-хеша. `10 TRX` ≈ пара долларов, не ~$10.
- Чем грозит (станет CRITICAL при включении): заплативший USDT не активируется — деньги застревают на адресе; любой адрес, случайно накопивший ≥10 TRX (пыль, ошибочный перевод), активирует лицензию; переплата принимается без записи.
- Как чинить: сканировать TRC-20 переводы (`/v1/accounts/{addr}/transactions/trc20`), фильтровать по контракту USDT, сверять `to` и `value`, хранить реальный `transaction_id`.
- Трудозатраты: L

**[MONEY-2] Нет проверки финальности блока** — `payments.py:239-249`
- Что не так: watcher реагирует на первый же снимок баланса ≥ expected, без требования подтверждённого/солидифицированного блока.
- Чем грозит (при включении): неподтверждённый или откатанный перевод выдаёт постоянную лицензию за так и не проведённые деньги.
- Как чинить: учитывать только переводы в солидифицированном блоке (TronGrid `confirmed=true` / глубина ≥19 блоков).
- Трудозатраты: S-M

**[MONEY-3] Оплата, пришедшая после истечения счёта, теряется навсегда: деньги приняты, лицензии нет** — `payments.py:112-118, 219-224` (фильтр `expires_at > now`); сессия живёт 120 мин
- Что не так: адрес на блокчейне валиден вечно, а watcher смотрит только на непросроченные сессии. Платёж на 121-й минуте не сканируется никогда, статуса «expired» тоже нет — строка навсегда «pending».
- Чем грозит: гарантированные споры и реальная потеря денег пользователем при любом чуть запоздавшем переводе (частое явление в крипте).
- Как чинить: отдельный проход по просроченным-но-свежим сессиям (grace-окно ~24ч), активировать или поднимать алерт админу.
- Трудозатраты: M

**[MONEY-4] `confirm_license` неатомарен: сессия помечается confirmed ДО выдачи лицензии** — `payments.py:165-193`
- Что не так: сначала `UPDATE ... status='confirmed'`, затем отдельным шагом INSERT лицензии — без общей транзакции. Если процесс упадёт между шагами, сессия уже confirmed (watcher её больше не увидит), а лицензии нет.
- Чем грозит: сбой по времени тихо превращает оплатившего пользователя в неактивированного; восстановление — только вручную админом.
- Как чинить: обернуть UPDATE + INSERT в один `async with database.transaction()`; сначала идемпотентный INSERT лицензии, потом отметка сессии.
- Трудозатраты: S

**[MONEY-8] Ошибка создания платёжного счёта проглатывается — клиент не получает ничего; плюс отсутствует зависимость bip_utils** — `main.py:2681-2686` (`except Exception: pass`); `payments.py:67` (lazy-import); `requirements.txt` (нет `bip_utils`)
- Что не так: любая ошибка при создании инвойса (нет bip_utils, гонка hd_index, не задан seed) тонет молча — ни фрейма ошибки, ни лога. `bip_utils` отсутствует в requirements (там есть только неиспользуемый tronpy), так что при включении оплаты первый же invoice упадёт именно сюда.
- Чем грозит: после включения `PAYMENTS_ENABLED` сервер стартует нормально, watcher бежит, но ни один пользователь не может получить адрес для оплаты — и никто этого не видит.
- Как чинить: добавить `bip_utils==2.9.3` в requirements; заменить `pass` на отправку `{"type":"payment-error"}` + `log_event`; при `PAYMENTS_ENABLED` дёргать `_get_wallet()` на старте и падать громко.
- Трудозатраты: S

**[DB-5] Денежная арифметика ведётся во float** — `payments.py:232, 204, 242, 249`; тот же паттерн `main.py:1840`, `admin_api.py:973`

- **Ядро WS-сессий**: fan-out `Dict[str,List[WebSocket]]`, идемпотентный presence (online на первом, offline на последнем сокете), sweeper мёртвых сокетов, per-user `asyncio.Lock` на офлайн-доставку от гонок reconnect (`main.py:1550-1556`). Комментарии объясняют «почему».
- **E2E-шифрование** реально защищает содержимое 1:1 даже при имперсонации из SEC-2 — сервер спроектирован не видеть plaintext личных сообщений.
- **SQL-инъекций нет**: всё через параметризованный SQLAlchemy core; единственный f-string SQL интерполирует ограниченный integer (1-1440) и admin-only (`tools.py:228`).
- **Пароли**: PBKDF2-SHA256 150k итераций + `secrets.compare_digest` + совместимость с bcrypt, минимум 12 символов (`main.py:757-782`, `auth_api.py:80`).
- **JWT-гейтинг админки полон**: все ~71 маршрут `admin_api.py` под `Depends(require_admin)` (проверено программно) — слабое место в самом секрете, не в покрытии.
- **Redis-мост** — чистый самодостаточный модуль: ZSET+TTL global-online (нет вечно-мёртвых на падении ноды), origin-tagged pub/sub, graceful degradation при недоступном Redis.
- **Oracle SQL-tool** заперт: admin-only, только SELECT, чёрный список ключевых слов, таймаут 10с, лимит 100 строк.
- **Numeric(20,6)** для всех денежных колонок (правильный ТИП; беда лишь в float-арифметике вокруг — DB-5).
- **delete_room**: broadcast до удаления + каскад в одной транзакции (`rooms_api.py:561-579`).
- **Promo-путь**: `activate-promo` оборачивает mark-used + insert в реальную транзакцию, промокоды через CSPRNG.
- **payment_watcher** изолирован от сбоев (traceback + продолжение цикла), не роняет event loop; seed-гигиена корректная.
- **Тесты**: 149 честных зелёных, реальные assert'ы на статусы/строки БД/WS-фреймы; покрыты рисковые зоны (WS-сессии, presence, offline, license gate, rooms delete, security).
- **Процессная дисциплина**: CHANGELOG + AI_WORKLOG форсятся pre-commit хуком; `docs/ai_kb/README.md` разделяет user-facing и внутренние доки; Dockerfile хорошо прокомментирован.
- **DoS-гарды**: 1MB лимит WS-сообщения, батч телеметрии 500/×10, TTL офлайн call-сигналов.
