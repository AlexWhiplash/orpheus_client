`api.orpheus.click`). Сформировано по итогам device-тестов клиента. По каждой задаче —
что, зачем, приоритет и ссылка на детальную спеку.

## 1. Оплата лицензии — USDT-TRC20 (клиент ГОТОВ, ждёт бэкенд) ⭐ высокий
**Статус:** добавлено 05.07 · не начато.
**Что:** приём оплаты лицензии криптой (USDT в сети TRON), self-hosted, без
посредников. Клиентская часть (экран покупки, сервис) **уже написана и в репозитории**
— не хватает только серверной.
**Нужно на сервере:**
- Endpoints (контракт уже зашит в клиент): `GET /api/purchase/tariffs`,
  `POST /api/purchase/invoice`, `GET /api/purchase/status`.
- HD-кошелёк: **свежий TRON-адрес на каждый инвойс** (BIP-44 `m/44'/195'/0'/0/i`).
- **Вотчер через TronGrid** (свою ноду не поднимаем): ловит входящий TRC-20 Transfer
  USDT на адрес заказа, ждёт solidified → активирует лицензию для pubkey.
- **Антифрод** (обязательно): сверять адрес контракта USDT, сумму, идемпотентность по
  tx_hash, только на solidified, rate-limit, возможность отзыва лицензии + бан.
**Спека:** `SERVER_USDT_TRON_IMPL.md` (готовый имплементейшн-гайд с телами
запросов/ответов) + `USDT_TRON_SPEC.md` (обоснование + антифрод). Почему выбрали
USDT-TRON — `PAYMENTS_OPTIONS.md`.
**Зачем:** это монетизация; клиент уже ждёт.

## 2. Управление WS-сессиями + presence + доставка license-status (КОРЕНЬ проблем надёжности) ⭐ высокий
**Статус:** добавлено 05.07 · обновлено 07.07 · **СЕРВЕР СДЕЛАЛ 08.07** — ждёт device-верификации (клиент не менялся).
**Симптомы с device-теста (всё сюда сходится):**
- WS у Samsung молча рвётся: коннект живёт **~30–48с**, затем `onDone`; после обрыва
  реконнекты **падают мгновенно несколько минут** (сервер отказывает почти сразу), пока
  старая сессия не протухнет. В клиентском логе причины обрыва нет.
- Presence лагает / залипает: после kill+reconnect клиента собеседник долго видит его
  оффлайн, звонок не проходит, пока сессия не протухнет.
- **«Онлайн не поднимался после активации лицензии»** — до перезапуска приложения
  (у некоторых Samsung). Клиент теперь форсит реконнект после активации как прикрытие,
  но корректно это должен обрабатывать сервер.
- **[07.07] Активация лицензии не впускала в приложение.** Промокод активировался
  (`activate-promo` → ok, сервер списал лицензию), но `license-status: active` по WS
  **не приходил** — ни сразу после активации, ни на последующих реконнектах. Клиент ждал
  этот пуш и не пускал внутрь. Прикрыто на клиенте входом по HTTP-ok, но серверу нужно
  доставлять статус по WS (см. ниже).
- Ответ на звонок на локе иногда со 2-го захода (первый ловит флап WS).
**Нужно на сервере (гипотезы — подтвердить/опровергнуть):**
- **Ключевой вопрос:** закрывает ли сервер предыдущее WS-подключение при новом коннекте
  того же pubkey? У клиента ДВА изолята (main + push) → возможны 2 подключения на pubkey.
  Если сервер их «выбивает» по кругу — это и есть флап. Паттерн «коннект жил ~40с →
  обрыв → реконнекты падают мгновенно» указывает именно сюда. Решение: разрешить несколько
  живых подключений на pubkey (fan-out доставки) ЛИБО аккуратный takeover без пинг-понга.
- Активный keepalive + быстрая чистка мёртвых сессий (ping/pong, чтобы presence не
  залипал онлайн на мёртвом сокете).
- **Ребродкаст presence** при (ре)подключении и отключении пира.
- **[07.07] Доставка `license-status`:** (а) слать `license-status: active` **сразу
  после активации** лицензии (промокод/оплата), не дожидаясь нового коннекта; (б)
  **переотправлять актуальный `license-status` на каждом (ре)коннекте** WS — сейчас,
  судя по логам, статус уходит только на первый коннект сессии.
- После активации лицензии — сессия должна перерегистрироваться как «лицензирована/
  онлайн» (связано с «онлайн после активации»).
**Спека:** `BACKEND_TASK_WS_SESSIONS.md` (симптомы, гипотезы H1-H3, что менять §4,
сценарии приёмки, контракт клиента: типы сообщений, keepalive, HTTP-fallback).
**Зачем:** это корень почти всех проблем со звонками/онлайном/лицензией в device-тесте.

## 3. Поддержка ДО активации лицензии (pre-auth) — небольшой
**Статус:** добавлено 05.07 · **СЕРВЕР СДЕЛАЛ 08.07** — ждёт e2e-проверки (клиент уже готов).
**Что:** endpoints поддержки должны принимать запрос от pubkey **без активной лицензии**.
**Нужно:** `POST /api/support/message` (+ `/messages`, `/unread`, `/logs`) — снять гейт
по лицензии, идентификация по заголовку `X-Pubkey`. Приложить введённый код/версию к
сообщению не обязательно, но полезно для контекста.
**Зачем:** кейс «купил лицензию, не активируется — некому написать». Клиент уже выводит
кнопку «Поддержка» на стартовый экран (до активации).
**Спека:** `ROADMAP.md`, пункт 1.

## 5. Удаление комнаты у всех — небольшой feature
**Статус:** добавлено 05.07 · **СЕРВЕР СДЕЛАЛ 08.07** · **КЛИЕНТ РЕАЛИЗОВАН** (сборка 1.1.7+19) — ждёт device-проверки.
**Что:** сейчас на клиенте есть только `leave` (выйти), `panic-clear` (очистить историю
у всех) и `rotate-invite`. Endpoint «удалить комнату целиком» **отсутствует**.
**Нужно:** `POST /api/rooms/{id}/delete` — **только для владельца** (`is_owner`):
удаляет комнату у всех участников.
**Зачем:** владелец хочет иметь возможность удалить свою комнату (не только выйти).
После появления endpoint добавлю на клиент пункт «Удалить комнату» (виден только владельцу).
**Спека:** `ROADMAP.md` (раздел серверных задач).
