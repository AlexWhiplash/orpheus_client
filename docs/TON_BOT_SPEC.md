# Orpheus — покупка лицензии через TON-бот: ТЗ

Реализация Этапа 1 оплаты в приложении (см. `PAYMENTS_OPTIONS.md`, вариант A).
Роли: **бэкенд+бот** — коллега-кодер (Python); **клиент** (Flutter) — я.

---

## 1. Идея и принцип

Оплата TON через Telegram-бота, лицензия привязывается к **публичному ключу**
пользователя. Способ оплаты отделён от выдачи лицензии: бот/бэкенд после оплаты
просто **активирует лицензию для pubkey** (переиспользуем существующий механизм
`activate-promo`/`license-status`). Клиент лишь показывает статус.

**Авто-активация без ручного кода** через одноразовый `session_token`:
1. Приложение просит у бэкенда `session_token` (привязан к pubkey).
2. Открывает бота с этим токеном в диплинке.
3. Пользователь платит TON.
4. Бот/бэкенд связывает `session_token → pubkey` → активирует лицензию.
5. Приложение ловит `license-status: active` (WS или поллинг) → само разблокируется.
Ручной код остаётся запасным путём (бот присылает код).

## 2. Способ оплаты TON — выбор (важно)

Все три ведут TON, но по-разному по суверенности:

- **T1. Прямой перевод TON на свой адрес + вотчер блокчейна** — ⭐ рекомендую по духу.
  Бот показывает: адрес кошелька + сумму + **comment/memo = order_id**. Пользователь
  шлёт из любого кошелька (Tonkeeper и т.п.). Бэкенд-вотчер (toncenter/tonapi) ловит
  входящую транзакцию с этим memo → матчит заказ. Нет третьих сторон, свой кошелёк,
  анонимно. Инфраструктура — просто вотчер.
- **T2. TON Connect** — подключение внешнего кошелька к боту/приложению и запрос
  транзакции (send N TON на наш адрес с payload). Тоже свой адрес, чуть глаже UX,
  но сложнее интеграция. Суверенно.
- **T3. Telegram Wallet Pay (@wallet)** — приём TON/USDT через сервис Wallet.
  Простейший UX, но добавляет Wallet как сторону (полу-кастодиально). Как опция.
- **❌ Telegram Stars — НЕ используем.** Stars покупаются юзером через Apple/Google
  IAP → ломается де-Google + ~30% комиссия сторов. Против концепции.

**Рекомендация:** старт на **T1 (прямой TON + вотчер)** — суверенно, анонимно,
минимум инфраструктуры. При желании добавить T2 (TON Connect) для гладкости.

## 4. Бэкенд — что нужно (кодеру)

**Endpoints:**
- `POST /api/purchase/session` — вход: `pubkey` (+ подпись, чтобы нельзя было
  создать сессию за чужой ключ). Выход: `{ session_token, deeplink, tariffs[] }`.
  `session_token` TTL ~30 мин, одноразовый.
- `GET /api/purchase/status?token=...` — (или переиспользовать `license-status`)
  для поллинга клиентом. Значения: `pending | paid | activated | expired`.
- (сущ.) `POST /api/activate-promo` — оставить как запасной ручной путь.

**Telegram-бот (aiogram/python-telegram-bot):**
- `/start <session_token>` → достаёт привязку token→pubkey → показывает тарифы
  (inline-кнопки).
- Выбор тарифа → создаёт `order` (order_id, tariff, TON-сумма по текущему курсу,
  фикс на N минут, `memo=order_id`) → показывает адрес + сумму + memo (или кнопку
  TON Connect).
- **Вотчер TON:** сервис/крон, опрашивающий toncenter/tonapi по нашему адресу;
  при входящей tx с известным memo и корректной суммой → помечает order `paid` →
  активирует лицензию для pubkey → шлёт WS-событие клиенту + сообщение в боте
  («Оплачено, лицензия активна») + запасной код.

**Конфиг/логика:**
- Тарифы и цены (в фиате или сразу в TON — см. §6).
- Идемпотентность: одна tx = одна активация; повтор не удваивает.
- Обработка недоплаты/просрочки/переплаты (см. §6).
- Курс: провайдер курса TON (если цена в фиате), фикс на срок order.

## 5. Клиент — что делаю я (Flutter)

- Кнопка **«Купить лицензию»**:
  - на `license_screen` (до активации) — главный кейс;
  - и в настройках (для продления).
- Запрос `POST /api/purchase/session` (с pubkey + подпись) → получить `deeplink`.
- Открыть диплинк бота (`url_launcher`), обработать отсутствие Telegram (fallback:
  показать инструкцию / ссылку).
- **Поллинг** `/api/purchase/status` (или слушать WS `license-status: active`) пока
  экран открыт → при `active` авто-разблокировка (как сейчас после `activate-promo`).
- Экран ожидания оплаты: «Ожидаем оплату…», кнопки «Открыть бота ещё раз» и
  «У меня есть код» (запасной ручной ввод — уже есть).
- Тарифы можно показать и в приложении (из ответа `session`), но оплату — в боте.
- Локализация RU/EN.

Клиентский контракт минимальный и не зависит от того, какой из T1/T2/T3 внутри —
приложение только открывает диплинк и ждёт статус.

## 6. Крайние случаи / решения

- **Курс TON плавает:** цену в фиате фиксируем в TON на срок order (напр. 15 мин);
  просрочил — новый order. Либо ценник СРАЗУ в TON (проще, без курс-провайдера).
- **Недоплата/переплата:** недоплату не активируем (сообщаем в боте, ждём допла́ту
  или возврат вручную); переплату — активируем (излишек — на усмотрение, обычно
  оставляем/возврат по запросу).
- **Просрочка order/сессии:** статус `expired`, предложить начать заново.
- **Юзер закрыл приложение после оплаты:** активация всё равно происходит (привязана
  к pubkey) — приложение подхватит `active` при следующем запуске.
- **Идемпотентность:** order_id/memo уникальны; повторная обработка tx — no-op.
- **Возвраты:** вручную через поддержку (крипта необратима технически).
- **Безопасность сессии:** `POST /purchase/session` подписывать ключом клиента,
  чтобы нельзя было создать сессию/активацию за чужой pubkey.

## 7. Этапность
1. **MVP:** T1 (прямой TON + вотчер) + endpoints + бот + клиентская кнопка/поллинг.
   Ценник сразу в TON (без курс-провайдера) для простоты.
2. **Полировка:** цена в фиате с фиксацией курса; TON Connect (T2) для гладкого UX;
   тарифы 1мес/6мес/бессрочно; продление из настроек.
3. **Позже (Этап 2 оплаты):** крипта через BTCPay (XMR + Lightning) для максимума
   анонимности — отдельный документ.
