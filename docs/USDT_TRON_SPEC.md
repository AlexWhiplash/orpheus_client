# Orpheus — оплата лицензии: self-hosted USDT-TRC20 (TRON)

**ВЫБРАННЫЙ основной способ оплаты (Этап 1).** In-app, без мессенджеров, self-hosted,
non-custodial по детекту. Заменяет ранее набросанный TON-бот (`TON_BOT_SPEC.md` —
устарел; TON можно добавить позже той же схемой). Роли: **бэкенд+вотчер** — кодер
(Python); **клиент** (Flutter) — я.

⚠ Значения помечены «проверить» — сверить перед боем (адрес контракта, лимиты TronGrid).

---

## 1. Почему USDT-TRC20 + self-hosted HD-вотчер
- **USDT — стейблкоин** (≈$1): фиксированный ценник, НЕ надо ловить/фиксировать курс.
- **TRON:** комиссии копеечные, блок ~3с, финализация ~1 мин.
- **USDT широко распространён** — у большинства уже есть, низкий порог.
- **Self-hosted:** свой приёмный HD-кошелёк + свой вотчер чейна. Нет процессора,
  нет KYC, нет мессенджера. Свежий адрес на КАЖДЫЙ инвойс → привязка к заказу без
  memo (каждый адрес = один инвойс).

---

## 2. Паттерн (суть)
1. HD-кошелёк (BIP-32/44) деривит **свежий TRON-адрес на каждый инвойс**
   (путь `m/44'/195'/0'/0/index`, TRON coin type = 195).
2. Инвойс = {order_id, pubkey, address, amount (USDT), expires}.
3. Вотчер следит за чейном: входящий **TRC-20 Transfer USDT** на адрес инвойса.
4. Достаточно подтверждений (solidified) + сумма совпала → активируем премиум
   (лицензию) для pubkey.
5. Приложение подхватывает статус `active` → разблокировано.

## 4. Бэкенд — что нужно (кодеру)

**HD-кошелёк / адреса:**
- Seed (BIP-39) → BIP-44 `m/44'/195'/0'/0/i`, инкремент `i` на каждый инвойс.
- **Non-custodial по детекту:** для ДЕТЕКТА достаточно публичной части (xpub) —
  деривим адреса, приватные ключи для этого не нужны. Приватные ключи (seed) нужны
  ТОЛЬКО для sweep (вывода средств) — держать офлайн/в секрете (см. §6).

**Endpoints:**
- `GET /api/purchase/tariffs` — тарифы **с ценами С СЕРВЕРА** (клиент цены НЕ
  хардкодит, меняются без обновления приложения):
  `{ tariffs: [{ id, title, duration_days, price_usdt }] }`.
- `POST /api/purchase/invoice` — вход: `{ pubkey, tariff_id }`. Выход:
  `{ order_id, address, amount_usdt, network: "TRC20", expires_at }`.
  (Подпись ключом клиента — опционально, только как антиспам; для MVP не обязательна,
  см. §Антифрод — привязка к адресу и так защищает.)
- `GET /api/purchase/status?order_id=...` — `pending | seen | confirmed | expired`.
- WS-событие `payment-confirmed` / `license-status: active` — **уже слушается клиентом**
  (`license_screen`), даёт авто-разблокировку.
- (сущ.) `POST /api/activate-promo` — запасной ручной путь остаётся.

**Вотчер TRON (через TronGrid — свою ноду НЕ поднимаем, решение владельца):**
- **TronGrid** = публичный API-шлюз к блокчейну TRON от TRON Foundation. Даёт HTTP/gRPC
  доступ к full-нодам БЕЗ поднятия своей. Mainnet: `https://api.trongrid.io`
  (тестнеты: Shasta/Nile). Бесплатный тариф по API-ключу (регистрация на trongrid.io),
  есть лимиты (порядка ~десятков запросов/сек + дневная квота — ⚠ уточнить актуальные;
  есть платные планы). Это **READ-зависимость, НЕ кастодиал** — TronGrid читает чейн,
  средствами не владеет.
- **Как ловим оплату:** поллинг по каждому активному адресу инвойса —
  `GET /v1/accounts/{address}/transactions/trc20?contract_address=<USDT>&only_to=true&min_timestamp=<t>`
  → входящие TRC-20 переводы USDT на адрес. Контракт USDT (mainnet)
  `TR7NHqjeKQxGTCi8q8ZY4pL8otSzgjLj6t` (⚠ проверить). USDT decimals = **6**.
- **Подтверждения:** ждать **solidified** блок (TRON финализирует ~19 блоков ≈1 мин).
  Активировать только после (не на «seen» в мемпуле).
- **Надёжность:** TronGrid может ратэ-лимитить/падать → вотчер с ретраями+backoff и
  **реконсиляцией по `min_timestamp`** (после простоя добрать пропущенное окно — платёж
  не потеряется). Фолбэк-провайдеры: GetBlock/NowNodes/Tatum/QuickNode.
- **Приватность (нюанс):** TronGrid видит, какие адреса мы опрашиваем (лёгкая утечка
  наших адресов третьей стороне), но не суммы владения и не средства. Приемлемо для
  MVP; при желании позже — свой java-tron нод или запросы через прокси.
- Идемпотентность: одна tx (по tx_hash) → одна активация; повтор — no-op.

**Активация:** сумма ≥ ожидаемой И solidified → пометить лицензию активной для pubkey
(тот же контракт, что `activate-promo`) → WS-событие клиенту + запасной код.

## 5. Клиент — что делаю я (Flutter)
- Кнопка **«Купить лицензию / Premium»**: на `license_screen` (до активации) + в
  настройках (продление).
- `POST /api/purchase/invoice` (pubkey+подпись) → получить `{address, amount, expires}`.
- **Экран оплаты (в приложении):**
  - Крупно: **сумма в USDT** + сеть «USDT (TRC-20 / TRON)» (важно — предупредить, что
    именно TRC-20, не ERC-20).
  - **QR** адреса (+ сумма, если кошелёк поддерживает) + адрес текстом + кнопка
    «Копировать».
  - Кнопка «Открыть кошелёк» (deep-link в TronLink/кошелёк, если поддерживается; иначе
    инструкция «отсканируйте QR в своём USDT-TRC20 кошельке»).
  - Таймер до `expires_at`.
  - **Поллинг** `/api/purchase/status` (или WS) → при `confirmed`/`active` — авто-
    разблокировка (как после `activate-promo`).
  - Кнопки «Я оплатил(а)» (форс-проверка), «У меня есть код» (ручной ввод — есть).
- Локализация RU/EN. Предупреждения: слать ТОЧНУЮ сумму, именно на TRC-20.

Клиентский контракт простой: открыть инвойс → показать адрес/QR/сумму → ждать статус.

## 6. Безопасность и sweep
- **Приватные ключи (seed) — не на боевом сервере в идеале.** Детект идёт по xpub
  (watch-only). Для **sweep** (вывода USDT с адресов инвойсов на основной кошелёк)
  нужны приватные ключи → делать отдельным процессом/офлайн, батчами, по расписанию.
  Это минимизирует hot-wallet риск (в отличие от «seed постоянно на сервере»).
- **TRX для sweep:** перевод TRC-20 требует energy/bandwidth (TRX) на адресе. Значит
  для sweep каждый адрес надо чуть пополнить TRX ИЛИ использовать делегирование
  energy. Операционная деталь бэкенда.
- Подпись `POST /purchase/invoice` ключом клиента — чтобы инвойс/активация не за чужой
  pubkey.

## 7. Крайние случаи
- **Недоплата:** не активируем; показать «получено X, нужно Y», ждать доплату/возврат.
- **Переплата:** активируем (излишек остаётся/возврат по запросу).
- **Просрочка инвойса:** статус `expired`, предложить новый инвойс (новый адрес).
  Поздняя оплата на старый адрес → обработать вручную / зачесть.
- **Неверная сеть (прислали ERC-20 USDT):** предупреждать в UI ЗАРАНЕЕ; такие
  средства на TRON-адрес не придут — это риск потери у пользователя, поэтому крупное
  предупреждение «только TRC-20 / TRON».
- **Подтверждения:** активируем только на solidified.

## 8. Ценник
- Цены **берём С СЕРВЕРА** (`GET /purchase/tariffs`) — клиент их **НЕ хардкодит**,
  можно менять без обновления приложения (решение владельца). В **USDT** (стейбл, без
  курс-провайдера). Тарифы/сроки (1мес/6мес/бессрочно) настраиваются на сервере.
- Клиент только отображает то, что вернул сервер, и сумму инвойса. Сумму для сверки
  оплаты знает СЕРВЕР (не доверяем клиентскому вводу) — см. §Антифрод.

## 8а. Антифрод (продумано)

Активация управляется ТОЛЬКО сервером по данным блокчейна — клиент не может «сказать
что оплатил». Основные векторы и защита:

1. **Фейковый токен под видом USDT** (шлют свой контракт с тикером USDT) → **сверять
   адрес контракта** ровно с настоящим USDT-TRC20. КРИТИЧНО. Иначе — бесплатная «оплата».
2. **Недоплата** → активируем только если `received >= amount` инвойса (USDT, 6 знаков).
3. **Двойной клейм / реплей одной оплаты** → свежий адрес на инвойс (адрес = один заказ)
   + идемпотентность по `tx_hash` (потрачен → больше не активирует).
4. **Оплата на просроченный инвойс** → не авто-активировать `expired`; в ручной разбор
   (зачесть/вернуть). Позднюю оплату не терять (реконсиляция вотчера).
5. **Неподтверждённая/reverted tx** → активировать только на **solidified** (§4).
6. **Клиент лжёт про оплату** → невозможно: активирует ТОЛЬКО вотчер по чейну; кнопка
   «Я оплатил» лишь форсит перепроверку статуса, не активацию.
7. **Матчинг по адресу, не по сумме** → т.к. адрес уникален на инвойс, привязка
   железная; сумма — вторичная проверка (не источник матча).
8. **Спам инвойсов** (исчерпание HD-индексов / замусоривание БД) → **rate-limit**
   `POST /purchase/invoice` по pubkey/IP; TTL инвойса; переиспользовать неоплаченные.
9. **Адрес НИКОГДА не переиспользуется** между инвойсами (даже после истечения лицензии).
10. **Фолбэк-рычаг:** сервер может **отозвать лицензию + забанить** pubkey/tx при
    выявленном абьюзе (решение владельца — крипта необратима, но лицензию отзываем).
11. **Мониторинг/алерты:** аномалии (много инвойсов, несовпадение сумм, оплаты на
    expired, чужой контракт) — флажить.

Замечание: подпись `POST /invoice` ключом клиента НЕ обязательна для корректности
(оплата на адрес инвойса активирует именно тот pubkey, что в заказе; «оплатить за
чужого» = подарить ему лицензию, не атака). Подпись — лишь опциональный антиспам.
